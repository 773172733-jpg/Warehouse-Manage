# 阶段2C4B：库存操作页面与流水查询部署指南

## 1. 变更范围

本阶段复用阶段2C4A已有的真实库存事务，完善统一库存操作页面，并新增按产品查看的永久库存流水页面。

- 统一库存操作页：`miniprogram/pages/stock-operation/`
- 产品库存流水页：`miniprogram/pages/stock-records/`
- 产品详情入口：入库、出库、盘点、库存记录
- 新增只读 action：`stock.records.list`
- 业务云函数仍为：`warehouse-api`

本阶段没有新增集合、数据库索引或数据库权限，没有修改库存事务核心计算、产品搜索规则或图片清理 Worker。

## 2. 库存操作页面

页面进入后使用 `warehouseProductId` 调用现有产品详情接口，重新读取产品、当前库存、最低库存、库存状态、`stockVersion` 和封面。跳转参数中的库存数量或版本不会被信任。

入库字段：

- 入库数量：大于0的整数
- 原因：可选，最长100字符
- 单据号：可选，最长50字符
- 当前库存与操作后库存预览

出库字段：

- 出库数量：大于0且不超过当前库存的整数
- 原因：可选，最长100字符
- 单据号：可选，最长50字符
- 当前库存与操作后库存预览

盘点字段：

- 盘点后实际库存：0至999999999的整数
- 调整原因：必填，最长100字符
- 单据号：可选，最长50字符
- 当前库存、盘点库存与调整差额预览

前端校验只改善操作体验，云端事务仍是最终判断。成功反馈展示云端返回的 `beforeStock`、`afterStock`、`delta`、库存状态和最新 `stockVersion`。

## 3. 重复提交与版本冲突

统一页面继续复用现有 requestKey 机制：

- 同一组业务参数重试时复用同一个 requestKey。
- 数量、原因、单据号或版本变化时生成新的 requestKey。
- requestKey 不写入 Storage，不输出到日志。
- 提交期间启用 saving 锁，连续点击不会发起第二次调用。
- 云端成功后，本页面进入已完成状态，不能再次执行同一变化。
- 页面卸载后不再执行 `setData`。

收到 `STOCK_VERSION_CONFLICT` 时，页面提示“库存已被其他操作更新，请刷新后重试”，重新读取当前库存与 `stockVersion`，保留用户输入，不自动再次提交。

## 4. 流水查询 Action

Action：`stock.records.list`

输入：

```json
{
  "warehouseProductId": "warehouse_product_xxx",
  "type": "all",
  "cursor": "可选稳定游标",
  "pageSize": 20
}
```

规则：

- `type` 支持 `all`、`initial`、`inbound`、`outbound`、`adjustment`。
- `pageSize` 默认20，最小1，最大50。
- `cursor` 与当前 `warehouseProductId` 和类型筛选绑定，筛选变化后不能复用。
- 服务端从可信用户上下文读取当前团队和仓库，不接受客户端传入 `teamId`、`warehouseId` 或身份字段。
- owner、admin、viewer 可以读取；pending、removed 或无有效团队成员关系的用户不能读取。
- 目标 `warehouseProduct` 必须属于当前团队、当前仓库并处于有效状态。

输出：

```json
{
  "items": [
    {
      "id": "stock_record_xxx",
      "type": "inbound",
      "beforeStock": 10,
      "afterStock": 15,
      "delta": 5,
      "quantity": 5,
      "reason": "补货",
      "referenceNo": "IN-20260718-001",
      "operatorDisplayName": "管理员",
      "operatorRole": "admin",
      "stockVersionBefore": 4,
      "stockVersionAfter": 5,
      "createdAt": "云数据库日期"
    }
  ],
  "nextCursor": "下一页游标或null",
  "hasMore": true,
  "pageSize": 20
}
```

响应不会返回 `requestHash`、完整 requestKey、`operatorUserId`、`teamId` 或其他完整身份和内部审计字段。

## 5. 分页与现有索引

流水固定按 `createdAt desc, _id desc` 排序，游标至少包含 `createdAt` 和 `_id`，并绑定当前查询条件。

本阶段复用现有索引 `idx_records_wh_product_created`：

1. `teamId asc`
2. `warehouseId asc`
3. `warehouseProductId asc`
4. `createdAt desc`
5. `_id desc`

不新增索引。类型筛选在云函数内使用上述稳定源游标进行有上限的分段扫描，每次 action 最多扫描500条源流水；不会一次读取全部流水，也不会把全量流水交给前端分页。若某一类型在超长历史中非常稀疏，页面可能需要多次“加载更多”才能继续越过无匹配区间。

## 6. 权限

- owner：入库、出库、盘点、查看流水。
- admin：入库、出库、盘点、查看流水。
- viewer：不显示库存操作入口，直接进入操作页会被阻止；可以查看当前团队当前仓库内有权限产品的流水。
- pending、removed：不能操作，也不能查看流水。

写操作继续由既有 `stock.inbound`、`stock.outbound` 和 `stock.adjust` 在服务端执行 owner/admin 权限校验。`stock.records.list` 只读，不提供 update、remove、撤销或批量删除入口。

## 8. 手动验收

入库：

1. owner和admin分别打开有库存产品详情。
2. 进入“入库”，输入正整数、可选原因和单据号。
3. 确认操作前后预览正确。
4. 提交后核对真实返回的前库存、后库存、变化量、状态和版本。
5. 验证0、负数、小数、非法字符和超过999999999上限的输入不能提交。

出库：

1. 输入不超过当前库存的正整数并提交。
2. 核对真实结果和产品详情自动刷新。
3. 验证超过当前库存时显示明确提示，且库存不变、不产生负库存。

盘点：

1. 分别测试调高、调低和调整为0。
2. 核对差额的正负号。
3. 验证原因必填，目标与当前库存相同时不能提交。

重复提交和冲突：

1. 连续点击确认，确认只产生一条流水。
2. 模拟弱网或超时后使用相同参数重试，确认获得幂等结果而不是重复变更。
3. 两个终端同时打开同一产品，先在终端A操作，再在终端B提交。
4. 确认终端B收到版本冲突提示、库存与版本刷新、原输入保留，且不会自动重试。

流水：

1. 从产品详情进入“库存记录”。
2. 分别使用 owner、admin、viewer 查看。
3. 核对 initial、inbound、outbound、adjustment 的数量、前后库存、原因、单据号、时间和角色。
4. 切换类型筛选，确认列表清空并重新加载，不混入上一筛选。
5. 连续加载更多，确认无重复、无跨产品记录。
6. 使用其他团队或其他仓库的 `warehouseProductId` 直接访问，确认被拒绝。

回归：

1. 搜索型号 `RA1809100` 的片段 `180`，确认仍可命中。
2. 验证名称搜索、分类和库存状态筛选。
3. 验证系统贴图、文字封面、自定义图片和私有临时URL。
4. 验证图片清理 Worker 行为不受影响。

以上为部署后的用户手动验收步骤；本地自动化检查不代表已完成真实微信开发者工具或真实 CloudBase 验收。

## 9. 回滚

代码回滚时，应按提交边界回滚2C4B的文档、流水 action 和页面变更，然后重新部署回滚后的 `warehouse-api` 并重新编译小程序。

不要删除或编辑已经产生的 `stock_records`。历史流水永久保留。不要使用 `reset --hard`、`clean` 或覆盖方式处理生产数据。

## 10. 已知限制与下一阶段建议

- 类型筛选不新增复合索引，因此使用最多500条源记录的有上限扫描；超长且某类型极稀疏的历史可能需要多次加载更多。
- 本阶段仅提供按产品的流水列表，不提供全仓真实流水总览、导出、冲销、采购、销售、供应商、客户、成本或财务能力。
- 现有底部“记录”页仍是旧展示范围，本阶段不将其改造成全仓真实流水页。

下一阶段可在独立审计索引、数据量和权限边界后，规划全仓流水总览或反向流水冲销。不得通过编辑或删除旧流水实现冲销。
