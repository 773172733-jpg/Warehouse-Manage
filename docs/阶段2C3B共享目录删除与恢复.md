# 阶段2C3B共享目录删除与恢复

## 两层回收语义

- 产品回收站：操作 `warehouse_products`。owner/admin可将零库存产品从当前仓库移除或恢复，`products`仍为active。
- 共享目录回收站：操作 `products`。仅owner可将团队共享产品目录软删除或恢复，所有 `warehouse_products` 和 `stock_records` 永久保留。

共享目录恢复只把 `products.status` 改回active，不会自动恢复任何仓库实例。产品仍不会出现在库存首页，owner/admin需要再到产品回收站恢复原 `warehouseProductId`。

## 接口与白名单

### product.catalog.delete

仅owner可调用。请求只允许：

- `productId`
- 正整数 `expectedVersion`
- 可选 `reason`，最多100字符
- `requestKey`

服务端先验证可信用户、active成员、owner角色和当前团队。随后读取products并检查activeWarehouseCount，再执行仓库实例定向查询；正式写事务会重新验证身份并锁定products与teams，products必须属于当前团队、status为active、version匹配且activeWarehouseCount严格为0。

服务端再执行一条按 `teamId + productId + _id` 定向的游标分页查询，逐页同时确认：

1. 不得存在 `status != removed` 的warehouse_products。
2. 不得存在 `stock != 0` 的warehouse_products。

activeWarehouseCount大于0返回 `PRODUCT_STILL_IN_WAREHOUSE`；计数为0但定向查询发现仓库状态矛盾时返回 `PRODUCT_WAREHOUSE_STATE_CONFLICT`，不会静默修正计数。

CloudBase文档型数据库事务当前只支持doc操作，不支持事务内where查询，见[官方事务限制](https://docs.cloudbase.net/database/transaction)。因此产品范围分页查询在事务紧邻之前执行，事务内再次读取并锁定products、重验version和activeWarehouseCount。所有受支持的仓库移除/恢复接口也会写同一products记录，若预检后发生合法并发，数据库写冲突会让事务回调重试并看到最新计数。不得绕过warehouse-api直接修改warehouse_products；控制台人工改数不属于并发保证范围。

成功后同事务将products改为deleted、version加1，写入可信deletedBy、deletedAt、deletionReason，并将teams.activeProductCount减1。warehouse_products和stock_records不写入、不删除。

### product.catalog.deleted.list

仅owner可读。请求只允许 `keyword`、`category`、`cursor`、`pageSize`。接口只查询可信当前团队的deleted products，默认20条、最大50条，按 `updatedAt desc, _id desc` 使用不透明游标。

搜索支持规范化编号精确匹配、规范化名称前缀和受控searchKeywords token，分类在云端筛选。响应不返回OPENID、内部用户/团队字段或幂等字段。

### product.catalog.restore

仅owner可调用。请求只允许 `productId`、正整数 `expectedVersion` 和 `requestKey`。事务前重复仓库状态定向检查；事务内要求products为deleted、version匹配、activeWarehouseCount仍为0。

teams.activeProductCount达到99,999时返回 `PRODUCT_LIMIT_REACHED`。成功后products改为active、version加1、目录删除当前状态字段清空，并写入可信restoredBy/restoredAt；teams.activeProductCount加1。不会创建或修改warehouse_products，也不会新增库存流水。

## 幂等与并发

删除和恢复分别保存最近成功意图的requestKey、规范化requestHash和结果version。哈希作用域包含action、可信teamId、productId和业务参数：

- 同键同参且目标仍处于该操作完成状态：返回原结果，`idempotent=true`，不重复修改version或计数。
- 同键异参：返回 `REQUEST_KEY_CONFLICT`。
- 非原请求对已deleted产品再次删除：返回 `PRODUCT_ALREADY_DELETED`。
- 非原请求对已active产品再次恢复：返回 `PRODUCT_ALREADY_ACTIVE`。

仓库移除、仓库恢复、主资料更新、目录删除和目录恢复都会在事务中读取并写入同一条products记录。数据库事务冲突重试使目录删除与仓库恢复不能同时成功：先提交目录删除时仓库恢复返回 `PRODUCT_CATALOG_DELETED`；先提交仓库恢复时目录删除看到activeWarehouseCount大于0并返回 `PRODUCT_STILL_IN_WAREHOUSE`。

## 页面权限

- owner：“我的”页同时显示产品回收站和共享目录回收站；产品回收站在后端返回 `canDeleteCatalog=true` 时显示“删除目录”。
- admin：可使用产品回收站恢复仓库实例，但看不到全局删除按钮和共享目录回收站入口。
- viewer：不显示任何回收站管理入口。
- pending/removed：不能读取产品业务数据。

共享目录页面和云端接口都会再次校验owner。页面不直调wx.cloud或数据库，不写Storage，不缓存完整数据库文档。

## 字段兼容

products按首次目录操作自然写入：

- `catalogDeleteRequestKey`、`catalogDeleteRequestHash`、`catalogDeleteResultVersion`
- `catalogRestoreRequestKey`、`catalogRestoreRequestHash`、`catalogRestoreResultVersion`
- `deletionReason`、`deletedBy`、`deletedAt`
- `restoredBy`、`restoredAt`

旧记录无需批量迁移。内部幂等和身份字段不返回客户端。

## 不变量

- 目录删除和恢复时products.activeWarehouseCount始终为0。
- teams.activeProductCount只在目录删除时减1、目录恢复时加1，且不小于0、不超过99,999。
- warehouse_products文档数量、ID、状态、库存和版本不因目录操作变化。
- stock_records数量、ID和内容不因目录操作变化。
- 目录恢复复用原productId；后续仓库恢复复用原warehouseProductId。
