# 阶段2B2A加入团队页面

## 申请人流程

无 active 团队的用户从团队创建页进入“加入已有团队”，输入 6 至 8 位邀请码。页面会去除空格、转换为大写并校验字符集，然后仅通过 `teamService.applyToJoin({ code, requestKey })` 提交。提交成功后立即调用 `teamService.getJoinStatus()`，以云端状态为准进入待审核页，不会直接进入库存页。

同一次申请因网络失败而重试时复用原 requestKey；用户修改邀请码后生成新的随机 requestKey。邀请码、requestKey 和申请状态均不写入 Storage，也不会记录到日志。

## 页面状态

- `input`：输入邀请码，可清除、粘贴、提交或返回创建团队。
- `submitting`：输入页按钮进入 loading 并禁用，防止连续点击。
- `pending`：只展示当前申请的团队名称、申请时间、待审核状态和刷新按钮。
- `rejected`：展示申请未通过；当前 2B1 响应未返回审核备注时显示“团队创建者未填写原因”。
- `removed`：提示原团队关系已移除，允许使用新邀请码重新申请。
- `approved`：刷新真实 bootstrap，成功后使用 `wx.switchTab` 进入库存页。
- `error`：状态读取或身份刷新失败，显示安全错误文案并允许用户主动重试。

pending 页面不提供撤销按钮，因为阶段2B1没有撤销申请接口。它也不提供返回库存或创建团队的入口，正常启动路径不能绕过审核。

## 启动路由

bootstrap 仍是主身份来源。有 active 团队时直接进入库存页；无 active 团队时才补充读取 `team.join.status`。pending、rejected 和 removed 进入加入团队页；没有申请进入创建团队页；join status 返回 active 时强制刷新 bootstrap 后再进入库存页。状态读取失败会停留在可重试错误页，不会无限重试或白屏。

team-setup 与 team-join 都会检查全局 bootstrap 身份；发现 active 团队后自动切换到库存 tab。当前业务 tab 仍以阶段1本地原型为主，本阶段保护正常 startup、team-setup 和 team-join 身份入口；后续云端产品库存接入时仍需为各业务页增加统一页面级鉴权。

## 隐私与服务边界

页面仅展示当前用户自己的申请和团队简要信息，不展示邀请码、OPENID、其他成员或内部用户记录。小程序端不直连数据库，也不直接调用 `wx.cloud.callFunction`；所有请求都经过既有 `cloud-service.js`。本阶段未修改 `warehouse-api`、集合、索引或数据库权限。
