# 阶段2C3B部署与验收

## 部署前确认

1. 确认当前项目为轻仓，正式AppID、CloudBase环境和云函数名与项目配置一致。
2. 确认products、warehouse_products、stock_records仍为客户端 `read=false`、`write=false`。
3. 本阶段不新增集合，不修改数据库权限。
4. 在 `warehouse_products` 人工创建以下两个普通索引，并等待状态变为可用：
   - `idx_wh_products_team_product_status`：`teamId`升序、`productId`升序、`status`升序；唯一关闭。
   - `idx_wh_products_team_product_stock`：`teamId`升序、`productId`升序、`stock`升序；唯一关闭。
5. 不要创建重复索引。products删除列表继续复用现有 `idx_products_team_status_updated`、分类、名称、编号和关键词索引。

两个新增索引分别支持“是否仍有未移除实例”和“是否仍有非零库存实例”的定向存在性检查，不对整个团队warehouse_products做全表扫描。

CloudBase事务不支持where查询，因此上述检查在写事务紧邻之前执行；写事务会再次锁定products并复核version与activeWarehouseCount。所有正式仓库移除/恢复也更新同一products记录，从而让合法并发通过事务写冲突重试互斥。禁止客户端或其他云函数绕过warehouse-api直接写warehouse_products。

## 必须重新部署

在微信开发者工具中右键 `cloudfunctions/warehouse-api`，选择“上传并部署：云端安装依赖”。本阶段新增三个action并修改产品模块，不重新部署就会得到 `UNKNOWN_ACTION` 或旧响应结构。

部署成功后重新编译小程序。不要创建第二个云函数，不要切换到其他小程序环境。

## 准备验收数据

1. 使用owner账号创建一个库存为0的测试产品，记录productId、warehouseProductId、products.version、teams.activeProductCount和stock_records数量/ID。
2. 从产品详情将它移出当前仓库。
3. 在云控制台确认warehouse_products仍存在、status=removed、stock=0，products.status仍为active且activeWarehouseCount=0。
4. 若团队未来有多个仓库，必须先在每个仓库移除该产品并确认所有实例stock均为0。

## 全局目录删除验收

1. owner从“我的”进入“产品回收站”。符合条件的产品显示“删除目录”；admin只显示“恢复”。
2. 点击“删除目录”，确认危险提示明确说明团队级软删除和永久保留历史流水，可选填写100字符以内原因。
3. 成功后云控制台确认products.status=deleted、version只加1、activeWarehouseCount仍为0，并有deletedAt/deletedBy/deletionReason。
4. 确认teams.activeProductCount只减1且不为负数。
5. 确认原warehouse_products文档仍存在，warehouseProductId、status、stock、stockVersion均未变化。
6. 确认stock_records数量、ID和内容与操作前完全一致。
7. 对activeWarehouseCount大于0的产品直接调用删除应返回 `PRODUCT_STILL_IN_WAREHOUSE`。
8. 制造“计数为0但存在active实例”或“removed实例stock非0”的测试脏数据时应返回 `PRODUCT_WAREHOUSE_STATE_CONFLICT`，验收后恢复测试数据，不在正式数据上长期保留脏状态。

## 共享目录回收站与恢复验收

1. owner从“我的”进入“共享目录回收站”，检查搜索、分类、下拉刷新、空状态、错误重试和cursor加载更多。
2. admin/viewer不应看到入口；手动编译页面路径也应被页面拦截，直接调用云函数应返回 `FORBIDDEN`。
3. 点击“恢复目录”，确认提示明确说明不会自动回到库存列表。
4. 成功后确认products复用原productId、status=active、version只加1、activeWarehouseCount仍为0，并有restoredAt/restoredBy。
5. 确认teams.activeProductCount只加1；达到99,999时恢复返回 `PRODUCT_LIMIT_REACHED`。
6. 返回库存首页，产品此时仍不应出现。
7. 进入产品回收站，恢复当前仓库实例；确认复用原warehouseProductId、stock=0、status=active，库存首页随后重新出现产品。
8. 再次确认stock_records没有新增、删除或修改。

## 幂等、版本与竞态验收

1. 删除或恢复请求网络超时后，使用相同requestKey和完全相同参数重试；version和teams.activeProductCount只能变化一次。
2. 使用同一requestKey改变reason、expectedVersion或productId，应返回 `REQUEST_KEY_CONFLICT`。
3. 两个owner页面同时打开同一目录项，一个先操作，另一个使用旧version操作，应返回 `PRODUCT_VERSION_CONFLICT`并刷新列表。
4. 目录删除与仓库恢复同时发起时只能有一个成功；最终不得出现products=deleted但warehouse_products=active。
5. 连续快速点击时当前项按钮应禁用，不得产生重复计数。

## 角色回归

- owner：可仓库移除/恢复、目录删除、查看和恢复共享目录。
- admin：可仓库移除/恢复，不可目录删除、查看或恢复共享目录。
- viewer：产品只读，不显示管理入口。
- pending/removed：不能读取产品业务数据。

## 阶段2C3A回归

重新验证product.create/list/detail/update、零库存仓库移除、正库存阻断、仓库回收站搜索/分页和原实例恢复。编辑version冲突必须继续生效；任何目录操作都不能修改库存余额或历史流水。

## 当前限制

- 微信开发者工具编译和真实CloudBase操作必须由用户手动完成。
- 本阶段不提供批量删除/恢复、目录合并、多仓切换UI、图片上传、库存写入或真实流水页面。
- 目录恢复后需要用户手动恢复仓库实例，这是两层生命周期的明确设计，不是缺陷。
