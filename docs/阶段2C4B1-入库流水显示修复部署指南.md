# 阶段2C4B1：入库流水显示修复部署指南

## 1. 真实根因

阶段2C4B的库存事务与流水查询字段一致：

- `stock.inbound` 在库存事务中真实写入 `stock_records`。
- 写入类型为 `inbound`。
- 写入和查询均使用当前团队、当前仓库和同一个 `warehouseProductId`。
- `createdAt` 使用数据库日期。
- `stock.records.list` 按 `createdAt desc, _id desc` 查询。
- 首次加载使用空 cursor，默认类型为 `all`。

缺陷位于小程序页面刷新生命周期。库存操作成功后原实现只设置库存首页刷新标记，`stock-records` 页面没有专用刷新标记、`onShow` 刷新或下拉刷新。若流水页面仍在页面栈中，重新显示时会继续使用旧列表，导致永久流水已经写入但页面暂时看不到。

本修复不通过前端插入临时流水，也不修改历史 `stock_records`。

- `miniprogram/pages/stock-operation/stock-operation.js`
- `miniprogram/pages/stock-records/stock-records.js`
- `miniprogram/pages/stock-records/stock-records.json`
- `tests/stage2c4b1.test.js`

## 3. 修复方式

库存操作成功后除原有首页刷新标记外，新增内存级流水刷新标记：

```js
stockRecordsRefreshRequired = {
  warehouseProductId
}
```

该标记不包含 requestKey、用户身份或完整流水内容，不写入 Storage。

库存流水页现在：

1. 首次进入时清空旧列表和 `nextCursor`，重新读取第一页。
2. 首次 `onShow` 不重复发起与 `onLoad` 相同的请求。
3. 后续 `onShow` 检测当前商品的刷新标记并重新读取产品摘要和第一页。
4. 不消费其他商品的刷新标记。
5. 支持下拉刷新，重新读取第一页。
6. 第一页请求进行中时合并刷新请求，避免并发覆盖。
7. 页面卸载后继续使用既有版本保护，不执行无效 `setData`。

## 4. 后端影响

### stock.inbound

不修改。库存和流水继续在同一数据库事务内原子提交。若 `stock_records` 写入失败，库存、库存状态和 `stockVersion` 均回滚。

### stock.records.list

不修改。继续使用可信用户上下文校验团队、仓库和 `warehouseProductId`，支持：

- `all`
- `initial`
- `inbound`
- `outbound`
- `adjustment`

排序仍为 `createdAt desc, _id desc`，默认 pageSize 为20，最大50。

## 5. 流水验证结果

自动化写后读测试确认：

- 普通入库成功后真实新增一条 `inbound` 流水。
- 流水包含正确的 `teamId`、`warehouseId`、`productId` 和 `warehouseProductId`。
- `beforeStock`、`afterStock`、`delta`、`quantity` 和前后 `stockVersion` 正确。
- `all` 和 `inbound` 筛选均能查询该流水。
- 最新入库流水排在第一页首项。
- 相同 requestKey 幂等重试不会增加第二条流水。
- 创建商品且初始库存大于0时真实新增 `initial` 流水。
- 初始库存为0时按现有规则不创建 initial 流水。
- 出库与盘点流水保持正常。

响应仍不返回 `requestHash`、完整 requestKey、`operatorUserId`、`teamId` 或 `warehouseId`。

## 8. 手动验收

### 普通入库

1. 使用 owner 或 admin 打开一个现有商品。
2. 记录当前库存，例如20。
3. 入库10，填写原因和单据号。
4. 确认成功结果为20到30、变化量 `+10`。
5. 返回商品详情。
6. 进入“库存记录 → 查看永久库存流水”。
7. 确认第一条为“入库 +10”，并显示20到30、原因、单据号、时间和角色。
8. 切换到“入库”筛选，确认同一条流水仍可见。

### 页面栈与刷新

1. 打开商品库存流水页。
2. 返回并完成一次库存操作。
3. 再次显示流水页，确认自动重新读取第一页。
4. 下拉刷新，确认最新流水仍排在顶部且没有重复。

### 新建商品初始库存

1. 新建商品并填写大于0的初始库存。
2. 进入该商品的永久库存流水。
3. 确认显示 `initial` 流水，操作前为0，操作后为初始库存。
4. 新建另一个初始库存为0的商品，确认按现有规则不生成 initial 流水。

### 权限与隔离

1. owner和admin确认可以入库并查看流水。
2. viewer确认不能入库，但可以查看当前团队当前仓库有权限商品的流水。
3. 确认其他团队或其他仓库的 `warehouseProductId` 不能读取。

以上步骤需要用户在微信开发者工具和正式 CloudBase 中完成，不应把本地自动化结果描述为正式部署或真实验收已完成。

## 9. 回滚

回滚本修复时只回滚2C4B1普通提交并重新编译小程序。不要：

- 删除或编辑已有 `stock_records`。
- 手工补写流水。
- 回滚库存数据或 `stockVersion`。
- 使用 `reset --hard` 或 `clean` 覆盖工作区。

## 10. 已知限制

- 本阶段仍是按单个 `warehouseProductId` 查看商品永久流水。
- 底部 Tab“记录”仍为旧预设页面，不是全仓真实流水总账。
- 本阶段不开发全仓总账、流水撤销、冲销、采购、销售或财务功能。
