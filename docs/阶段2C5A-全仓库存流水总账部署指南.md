# 阶段2C5A 全仓库存流水总账部署指南

本阶段把底部 Tab“记录”接入真实 `stock_records` 全仓库存流水总账。用户可以在当前可信团队与当前仓库范围内，只读查看初始库存、入库、出库和盘点调整记录。

## 1. 正式配置

- AppID：`wxd5819a772c90b7a2`
- CloudBase 环境：`cloud1-d8gm59cz2be4e7c23`
- 业务云函数：`warehouse-api`
- 禁止部署到事件管理器项目或其 CloudBase 环境

## 2. 新增 Action

新增云函数 Action：

```text
stock.records.listWarehouse
```

现有 `stock.records.list` 继续用于“单个商品详情 -> 库存记录”，本阶段不修改它的职责。

输入参数：

```json
{
  "type": "all | initial | inbound | outbound | adjustment",
  "startAt": "可选，ISO UTC 时间",
  "endAt": "可选，ISO UTC 时间",
  "cursor": "可选，服务端返回的游标",
  "pageSize": "可选，默认20，最大50"
}
```

客户端不得传入 `teamId`、`warehouseId`、`userId`、`operatorId` 或角色字段。服务端只从可信微信上下文、当前用户、当前团队成员关系和当前仓库状态确定查询范围。

## 3. 权限

- `owner`：可以查看当前团队、当前仓库的全仓流水。
- `admin`：可以查看当前团队、当前仓库的全仓流水。
- `viewer`：可以只读查看当前团队、当前仓库的全仓流水。
- `pending` / `removed` 成员：不可查看，返回无权错误。

本阶段不新增任何写入、删除、撤销、回滚或修改 `stock_records` 的接口。

## 4. 索引

本阶段需要 `stock_records` 存在以下等价索引，均为非唯一：

```text
idx_records_wh_created
teamId 升序
warehouseId 升序
createdAt 升序或降序
_id 升序或降序
非唯一
```

```text
idx_records_type_created
teamId 升序
warehouseId 升序
type 升序
createdAt 升序或降序
_id 升序或降序
非唯一
```

正式环境已由人工在 CloudBase 控制台补齐 `idx_records_type_created`。代码不会自动创建、删除或修改数据库索引。

## 5. 查询与分页

- 查询范围固定为当前可信 `teamId` 和当前有效 `warehouseId`。
- 排序固定为 `createdAt DESC, _id DESC`。
- `type=all` 使用全仓时间索引。
- 指定类型时使用 `teamId + warehouseId + type + createdAt + _id` 索引。
- 游标包含 `createdAt`、`_id`、查询条件哈希和分页版本。
- 切换类型、切换时间范围或改变 `pageSize` 后，旧游标会失效。
- 页面不会一次性读取全部流水，也不会在前端做全量本地分页。

## 6. 产品信息映射

接口按当前页批量读取 `products` 和 `warehouse_products`，合并商品名称、型号、单位和封面信息。已删除、移出或历史商品的流水仍然显示；无法继续查看的商品会保留历史流水，并在点击时提示“该商品当前不可查看”。

响应不会返回：

- `teamId`
- `warehouseId`
- `operatorUserId`
- `requestKey`
- `requestHash`
- 完整数据库记录
- 完整图片 fileID

图片封面继续使用现有私有图片临时 URL 逻辑，不直接拼接公开 URL。

## 7. 小程序页面

底部 Tab“记录”改为真实库存流水总账，支持：

- 类型筛选：全部、入库、出库、盘点、初始库存。
- 时间筛选：全部时间、今天、近7天、近30天。
- 下拉刷新。
- 触底或按钮加载更多。
- 加载、空状态、错误重试。
- 点击可查看的记录进入商品详情。

入库、出库、盘点成功后会标记全仓流水刷新；新增商品产生初始库存流水时也会标记总账刷新。

## 8. 部署步骤

1. 在微信开发者工具确认 AppID 是 `wxd5819a772c90b7a2`。
2. 在云开发控制台确认环境是 `cloud1-d8gm59cz2be4e7c23`。
3. 确认 `stock_records` 已存在 `idx_records_wh_created` 和 `idx_records_type_created`，均为非唯一。
4. 右键 `cloudfunctions/warehouse-api`。
5. 选择“上传并部署：云端安装依赖”。
6. 不部署 `product-image-cleanup-worker`。
7. 不修改数据库权限、云存储权限、AppID 或 CloudBase 环境。
8. 重新编译小程序，必要时清除编译缓存。

## 9. 手动验收

1. 使用 owner 或 admin 账号打开小程序。
2. 进入底部 Tab“记录”，确认显示真实库存流水，不再是预设假数据。
3. 新增一个有初始库存的商品，回到底部“记录”，确认出现“初始库存”记录。
4. 对已有商品执行入库，回到底部“记录”，确认出现入库记录，包含操作前、操作后、原因、单据号、时间和角色。
5. 执行出库和盘点调整，确认记录出现且类型筛选可命中。
6. 切换“今天 / 近7天 / 近30天 / 全部时间”，确认列表刷新且不会重复追加旧数据。
7. 点击有效商品的流水，确认能进入商品详情。
8. 对已移出或删除商品的历史流水，确认仍显示，点击时不白屏。
9. 使用 viewer 账号确认可以只读查看流水，不能进行库存操作。
10. 使用 pending 或 removed 成员确认不能查看流水。

## 10. 回滚

如需回滚，按提交边界回滚 2C5A 的云端 Action、记录 Tab 页面和本部署文档，然后重新部署上一版 `warehouse-api` 并重新编译小程序。回滚代码不会删除或修改 `stock_records` 历史流水。

## 11. 已知限制

- 本阶段只做全仓库存流水总账，不做采购单、销售单、客户供应商、成本或财务统计。
- 本阶段不做流水撤销、反向冲销或库存报表统计。
- 当前底部总账按当前仓库查看，不跨仓库汇总。
- 正式验收必须以真实微信开发者工具和正式 CloudBase 环境为准。
