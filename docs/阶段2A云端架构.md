# 阶段2A云端架构

## 环境边界

轻仓固定使用 AppID `wxd5819a772c90b7a2`、CloudBase 环境 `cloud1-d8gm59cz2be4e7c23` 和云函数 `warehouse-api`。小程序初始化和每次 `callFunction` 都显式指定该环境，服务层不允许覆盖云函数名称。云函数使用 `cloud.DYNAMIC_CURRENT_ENV`，部署时必须人工确认目标环境，代码不会回退到其他环境。

## 可信身份

`warehouse-api` 在每次请求中调用 `cloud.getWXContext()` 获取 OPENID。OPENID 仅用于云端用户定位，不写日志、不放进响应；客户端提交的 `openId`、`userId`、`role`、`teamId` 和 `warehouseId` 不参与身份或权限决策。

用户 `_id` 由 OPENID 的 SHA-256 摘要生成，重复或并发 bootstrap 会操作同一文档。`users.openId` 唯一索引提供第二层约束。

## 路由与响应

静态白名单只允许：

- `system.ping`
- `user.bootstrap`
- `team.create`
- `team.current`

响应统一包含 `success`、`requestId`、`timestamp`，成功返回 `data`，失败返回稳定的 `error.code` 和可展示信息。未知 action 返回 `UNKNOWN_ACTION`；数据库堆栈不会返回客户端。云端日志只记录 requestId、action 和错误码。

## 权限基础层

- `requireUser`：按可信 OPENID 查找 active 用户。
- `requireActiveMembership`：查询指定用户在服务端确定团队中的 active 关系。
- `requireRole`：通过显式等级 `owner > admin > viewer` 判断，不比较字符串大小。

页面按钮控制只改善体验，最终权限必须由云函数查询 `team_members` 决定。

## user.bootstrap

1. 获取可信 OPENID。
2. 在数据库事务内读取确定性用户文档。
3. 首次调用创建用户；再次调用只更新 `lastLoginAt` 和 `updatedAt`。
4. 查询 active 成员关系、active 团队和默认仓库。
5. 修复失效的当前团队或仓库指针。
6. 返回清洗后的用户、成员关系、团队、仓库和 `onboardingRequired`。

数据不一致时不伪造团队。无有效成员关系或团队时返回 `onboardingRequired: true`。

## team.create

只接收 `name`、`warehouseName`、`requestKey`。服务端清洗输入，忽略所有额外身份字段。

团队 ID 由可信用户 ID 和 requestKey 确定；默认仓库 ID、owner 成员关系 ID也为确定性 ID。创建流程在一次服务端事务内锁定用户文档并完成：团队、默认仓库、团队回写、owner 成员关系、用户当前团队和仓库指针。任一步失败则全部回滚。

同一 requestKey 重试读取同一组文档并返回已有完整结果。不同 requestKey 并发创建时都会更新同一用户文档，事务写冲突重试后只能有一个请求通过 V1 的单 active 团队检查。

## team.current

接口按可信当前用户查询 active 成员关系，只返回该用户自己的团队、角色和默认仓库。无团队是正常空状态；disabled 或 deleted 团队不会作为有效团队返回。

## 前端启动

`App.onLaunch` 初始化指定 CloudBase 环境并发起去重 bootstrap。startup 复用同一个 Promise：有团队用 `wx.switchTab` 进入库存页，无团队用 `wx.redirectTo` 进入团队初始化页，云函数失败则停留并显示可重试错误。

团队初始化页失败后保留输入和 requestKey；输入变化后才生成新 requestKey。创建成功直接更新全局 bootstrap 状态并进入库存页。

阶段1页面中的产品、库存、记录、团队成员和个人页业务数据仍为本地原型，本阶段不把这些模拟数据写入云端。
