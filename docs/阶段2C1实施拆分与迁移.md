## 2. 页面迁移原则

- 不一次性删除全部mock；按页面完成loading、空、错误、分页和权限后再切换。
- service层把products和warehouse_products组合成当前WXML需要的view model。
- 前端不自行访问或拼接两个集合。
- 客户端不保存全量产品，不用本地全量筛选代替云端搜索。
- 写操作成功优先使用接口返回的新版本更新当前页，再在onShow刷新。
- viewer隐藏写入口，但云函数仍独立拒绝。

## 3. 页面迁移

### 3.1 库存首页

1. `product.list`替换mock.getProducts，服务端返回当前仓库组合列表。
2. 搜索、分类和库存状态改为服务端参数；每页20至50条游标分页。
3. summary使用warehouse_products索引统计和短期缓存。
4. 区分首屏loading、空仓库、筛选无结果、失败重试。
5. 页面“删除产品”改为product.removeFromWarehouse。
6. 增加当前仓库回收站入口和restoreToWarehouse操作。

### 3.2 产品详情

1. URL只传productId，服务端组合主资料与当前仓库余额。
2. 使用真实stockStatus和最近流水。
3. warehouse_products removed时进入回收站详情语义；products deleted时提示目录产品不可用。
4. 快照版本落后由服务端合并权威主数据并修复。

### 3.3 产品创建

1. 先支持文字和emoji封面；图片后续独立接入。
2. 表单requestKey在输入不变的重试中复用。
3. product.create一次事务创建products、warehouse_products和可选initial流水。
4. 达到团队99,999 active产品时明确提示上限。
5. 成功后刷新首页并定位新仓库实例。

### 3.4 产品编辑

1. 编辑products主资料，携带version。
2. 不提交stock、minStock、stockStatus或stockVersion。
3. PRODUCT_VERSION_CONFLICT保留用户输入并要求刷新确认。
4. V1同步当前仓库快照；多仓时由版本检测和补偿完成其余同步。
5. 最低库存改由inventory.threshold.update处理。

### 3.5 仓库移除与恢复

1. 移除前页面可提示stock必须为0，服务端事务再次校验。
2. 移除只改变warehouse_products，不删除products或图片。
3. 回收站按removed状态分页。
4. 恢复复用原实例，stock为0，快照同步到最新产品版本。

### 3.6 库存操作

1. 页面进入时读取最新组合详情。
2. inbound/outbound/adjust分别调用真实接口并复用requestKey重试。
3. 出库不足、版本或权限变化时刷新。
4. 成功后使用返回stockVersion和流水更新页面。

### 3.7 流水页

1. `stock.record.list`替换mock，按createdAt游标分页。
2. 产品、类型、操作人、日期和关键词均由服务端筛选。
3. 详情只读，无修改或删除入口。
4. 产品、仓库实例或成员状态变化后仍显示历史快照。

## 4. 后续阶段拆分

### 2C2：共享目录、仓库实例和真实读取

范围：

- 人工创建products、warehouse_products、stock_records及2C2必需索引和权限。
- 为teams增加服务端activeProductCount兼容字段。
- 实现product.create/list/detail和initial流水事务。
- 接入库存首页和产品详情，先支持文字/emoji封面。
- 完成99,999上限、重名产品、分页、统计、快照版本校验。

独立验收：owner/admin创建；viewer只读；同名ID区分；初始库存和流水原子一致；首页不加载全量。

### 2C3A：编辑与仓库级回收站

范围：

- 实现product.update、removeFromWarehouse、restoreToWarehouse。
- 接入仓库产品回收站和恢复入口。
- 实现主数据快照同步与异步补偿基础。

独立验收：版本冲突、非零库存禁止移除、恢复复用原实例。

### 2C3B：全局共享产品目录回收站

范围：

- 实现product.catalog.delete、product.catalog.restore、product.catalog.listDeleted。
- 接入全局共享产品目录回收站UI。

全局目录删除条件：

1. 仅owner可执行。
2. 所有关联warehouse_products库存均为0。
3. 不存在active仓库实例。
4. products只进行软删除。
5. stock_records永久保留。
6. 恢复时复用原products记录。
7. 不重新创建产品ID。
8. V1确定需要该UI，但不阻塞2C2。

独立验收：owner专属权限、零库存校验、目录恢复不自动恢复仓库实例。

### 2C3C：自定义图片

范围：

- 接入最大2MB自定义图片prepare/upload/bind/discard。
- staging孤立文件24小时后清理。
- 旧图片延迟回收（无引用后清理）。

独立验收：2MB限制、伪造MIME拒绝、旧图与孤立文件生命周期。

### 2C4：真实库存事务

范围：

- 实现inbound、outbound、adjust、threshold.update。
- 使用warehouse_products.stockVersion、事务写冲突重试和requestKey幂等。
- 接入库存操作页。

独立验收：双账号并发、防超卖、同key重试、事务回滚、阈值边界。

### 2C5：真实流水、统计、搜索优化与99,999压测

范围：

- 接入流水列表、详情和组合筛选。
- 补齐流水、分类和库存状态索引。
- 验证名称前缀、编号、关键词搜索性能和首页统计缓存。
- 数据规模需要时评估专用全文索引或搜索服务，不承诺任意正则。
- 99,999规模专项压测（见第8节）。

独立验收：永久历史、删除/离队后的快照、组合筛选、跨页稳定性和统计一致性。

## 6. 测试账号与验收

至少使用同一团队内owner、admin、viewer三个真实账号，并用owner/admin并发操作同一仓库产品。

### 产品与目录

- 创建两个同名产品，确认以ID独立打开。
- 第99,999个active产品允许，第100,000个被拒绝；目录删除/恢复正确维护计数。
- 主资料更新后当前仓库快照同步；模拟旧productVersion时读取返回权威字段并修复。

### 移除与恢复

- 非零库存不能移除；归零后进入回收站。
- 恢复复用同一warehouseProductId，stock仍为0，历史流水不变。
- 目录deleted时仓库实例不能恢复；目录恢复不自动恢复仓库实例。

### 库存与流水

- 入库、出库、调整后余额和流水前后值一致。
- 并发出库最终stock不小于0。
- 网络超时同key重试只产生一条流水。
- 流水没有修改/删除能力，产品、仓库实例、成员变化后仍可查询。

### 图片

- 2MB以内合法图片成功，超2MB或伪造MIME被拒绝。
- 本地路径和临时URL不入库。
- 仓库移除、目录软删除不删除主图；旧图仅在无引用后延迟回收。
- 创建失败的staging文件按计划清理。

## 7. 已确认与未决事项

已确认，不再作为待定项：

- 每团队active共享产品逻辑上限99,999。
- 名称允许重名，以 `_id` 为唯一身份。
- 当前仓库回收站和恢复入口必须实现。
- 图片最大2MB。
- 流水永久保留，只允许未来冷归档。
- 共享产品目录确定需要，最终采用简化方案C。

以下由项目负责人最终确认（2C1文档封口）：

1. ✅ 全局目录删除与恢复UI安排在2C3B。V1确定需要该UI，但不阻塞2C2。仅owner可操作，要求所有关联warehouse_products库存为0且无active仓库实例。products只软删除，stock_records永久保留，恢复时复用原products记录不重新创建产品ID。
2. ✅ searchKeywords只能由warehouse-api服务端纯函数生成。product.create和product.update统一调用同一生成函数。前端不得生成、提交或覆盖searchKeywords；即使请求中出现也丢弃。批量导入、数据修复和管理工具必须复用同一生成函数。来源包括name、productCode、category、brand、specification，服务端负责去首尾空格、统一英文大小写、规范连续空白、去重、过滤空值、限制数量（最多10项）和长度（每项1至20字符）。客户端禁止提交searchKeywords、normalizedName、normalizedCode字段。
3. ✅ 99,999极限规模不在2C2做真实云端极限灌数测试。2C2必须完成游标分页（默认20/最大50）、事务限额、必要索引和禁止客户端全量加载。2C5在搜索、统计和索引完整后进行专项压测，至少覆盖99,999条products及对应warehouse_products下的首页分页、名称前缀查询、编号查询、分类筛选、库存状态筛选、首页统计和流水分页。压测使用专门测试团队或独立测试环境，不污染正式业务团队。发布候选版前根据压测结果决定是否需要专用搜索服务或统计汇总集合。

## 8. 99,999规模能力基线与专项压测

### 8.1 2C2必须完成的规模基础能力

2C2不进行99,999条真实云端极限灌数测试，但必须完成以下基础能力：

1. 服务端游标分页：每页默认20条，最多50条；游标由服务端编码，客户端原样回传。
2. 团队产品数量事务限额：teams.activeProductCount在创建products时事务性检查99,999上限。
3. 必要索引：2C2前创建的products和warehouse_products索引（见database/indexes.md）。
4. 严禁客户端全量加载：接口不提供一次性下载所有产品的能力，游标分页是唯一读取路径。
5. 首页统计使用索引count/聚合加短期缓存，不扫描全表。

### 8.2 2C5专项压测范围

2C5在搜索、统计和所有业务索引创建完成后执行专项压测，至少覆盖：

| 场景 | 数据规模 | 验证点 |
| --- | --- | --- |
| 首页分页 | 99,999条products + 对应warehouse_products | 默认20条游标响应时间、跨页稳定性 |
| 名称前缀查询 | 同上 | normalizedName范围查询性能 |
| 编号查询 | 同上 | normalizedCode精确/前缀匹配 |
| 分类筛选 | 同上 | categorySnapshot索引筛选 |
| 库存状态筛选 | 同上 | stockStatus索引筛选 + 分类组合 |
| 首页统计 | 同上 | total/low/out聚合响应时间，缓存命中 |
| 流水分页 | 99,999条products对应流水 | 按createdAt游标分页、类型筛选 |

### 8.3 压测环境与数据隔离

1. 压测必须使用专门测试团队或独立CloudBase测试环境。
2. 压测数据不得写入正式业务团队的products、warehouse_products或stock_records。
3. 压测账号不得与正式用户的OPENID、团队或仓库有任何关联。
4. 压测完成后清理测试环境和数据。

### 8.4 压测结果决策

1. 发布候选版前根据压测结果决定是否需要专用搜索服务（如CloudBase全文索引或Elasticsearch）。
2. 首页统计如聚合性能不足，评估预计算汇总集合（如warehouse_summary）。
3. 流水如出现跨页不稳定，评估复合索引或时间窗口分区策略。
4. 决策依据为p95响应时间而非平均值，因为分页场景更需要尾部延迟保证。
