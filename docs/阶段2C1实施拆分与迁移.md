# 阶段2C1实施拆分与迁移

## 1. 当前原型字段映射

## 2. 页面迁移原则

- 不一次性删除全部mock；按页面完成loading、空、错误、分页和权限后再切换。
- service层把products和warehouse_products组合成当前WXML需要的view model。
- 前端不自行访问或拼接两个集合。
- 客户端不保存全量产品，不用本地全量筛选代替云端搜索。
- 写操作成功优先使用接口返回的新版本更新当前页，再在onShow刷新。
- viewer隐藏写入口，但云函数仍独立拒绝。

## 3. 页面迁移

### 3.1 库存首页

1. `product.list`替换mock.getProducts，服务端返回当前仓库组合列表。
2. 搜索、分类和库存状态改为服务端参数；每页20至50条游标分页。
3. summary使用warehouse_products索引统计和短期缓存。
4. 区分首屏loading、空仓库、筛选无结果、失败重试。
5. 页面“删除产品”改为product.removeFromWarehouse。
6. 增加当前仓库回收站入口和restoreToWarehouse操作。

### 3.2 产品详情

1. URL只传productId，服务端组合主资料与当前仓库余额。
2. 使用真实stockStatus和最近流水。
3. warehouse_products removed时进入回收站详情语义；products deleted时提示目录产品不可用。
4. 快照版本落后由服务端合并权威主数据并修复。

### 3.3 产品创建

1. 先支持文字和emoji封面；图片后续独立接入。
2. 表单requestKey在输入不变的重试中复用。
3. product.create一次事务创建products、warehouse_products和可选initial流水。
4. 达到团队99,999 active产品时明确提示上限。
5. 成功后刷新首页并定位新仓库实例。

### 3.4 产品编辑

1. 编辑products主资料，携带version。
2. 不提交stock、minStock、stockStatus或stockVersion。
3. PRODUCT_VERSION_CONFLICT保留用户输入并要求刷新确认。
4. V1同步当前仓库快照；多仓时由版本检测和补偿完成其余同步。
5. 最低库存改由inventory.threshold.update处理。

### 3.5 仓库移除与恢复

1. 移除前页面可提示stock必须为0，服务端事务再次校验。
2. 移除只改变warehouse_products，不删除products或图片。
3. 回收站按removed状态分页。
4. 恢复复用原实例，stock为0，快照同步到最新产品版本。

### 3.6 库存操作

1. 页面进入时读取最新组合详情。
2. inbound/outbound/adjust分别调用真实接口并复用requestKey重试。
3. 出库不足、版本或权限变化时刷新。
4. 成功后使用返回stockVersion和流水更新页面。

### 3.7 流水页

1. `stock.record.list`替换mock，按createdAt游标分页。
2. 产品、类型、操作人、日期和关键词均由服务端筛选。
3. 详情只读，无修改或删除入口。
4. 产品、仓库实例或成员状态变化后仍显示历史快照。

## 4. 后续阶段拆分

范围：

- 人工创建products、warehouse_products、stock_records及2C2必需索引和权限。

独立验收：owner/admin创建；viewer只读；同名ID区分；初始库存和流水原子一致；首页不加载全量。

### 2C3：编辑、仓库移除恢复和图片

范围：

- 实现product.update、removeFromWarehouse、restoreToWarehouse。

独立验收：版本冲突、非零库存禁止移除、恢复复用原实例、旧图与孤立文件生命周期。

### 2C4：真实库存事务

范围：

- 实现inbound、outbound、adjust、threshold.update。
- 使用warehouse_products.stockVersion、事务写冲突重试和requestKey幂等。

独立验收：双账号并发、防超卖、同key重试、事务回滚、阈值边界。

### 2C5：真实流水、统计和搜索优化

范围：

- 接入流水列表、详情和组合筛选。

- 验证名称前缀、编号、关键词搜索性能和首页统计缓存。
- 数据规模需要时评估专用全文索引或搜索服务，不承诺任意正则。

独立验收：永久历史、删除/离队后的快照、组合筛选、跨页稳定性和统计一致性。

## 6. 测试账号与验收

至少使用同一团队内owner、admin、viewer三个真实账号，并用owner/admin并发操作同一仓库产品。

### 产品与目录

- 创建两个同名产品，确认以ID独立打开。
- 第99,999个active产品允许，第100,000个被拒绝；目录删除/恢复正确维护计数。
- 主资料更新后当前仓库快照同步；模拟旧productVersion时读取返回权威字段并修复。

### 移除与恢复

- 非零库存不能移除；归零后进入回收站。
- 恢复复用同一warehouseProductId，stock仍为0，历史流水不变。
- 目录deleted时仓库实例不能恢复；目录恢复不自动恢复仓库实例。

### 库存与流水

- 入库、出库、调整后余额和流水前后值一致。
- 并发出库最终stock不小于0。
- 网络超时同key重试只产生一条流水。
- 流水没有修改/删除能力，产品、仓库实例、成员变化后仍可查询。

### 图片

- 2MB以内合法图片成功，超2MB或伪造MIME被拒绝。
- 本地路径和临时URL不入库。
- 仓库移除、目录软删除不删除主图；旧图仅在无引用后延迟回收。
- 创建失败的staging文件按计划清理。

## 7. 已确认与未决事项

已确认，不再作为待定项：

- 每团队active共享产品逻辑上限99,999。
- 名称允许重名，以 `_id` 为唯一身份。
- 当前仓库回收站和恢复入口必须实现。
- 图片最大2MB。
- 流水永久保留，只允许未来冷归档。
- 共享产品目录确定需要，最终采用简化方案C。

进入实现前仍需负责人确认：

1. `product.catalog.delete/restore`全局目录管理UI安排在哪个后续版本。
2. 受控searchKeywords由用户填写、系统生成，还是两者合并。
3. 99,999极限规模是否需要在2C2进行压测，或先以代表性数据量验证查询计划。
