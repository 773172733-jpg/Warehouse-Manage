# 阶段2B2B2成员管理与退出团队

## 功能范围

团队页继续使用 `team.member.list` 返回的真实 active 成员，不读取 `mock-members.js`。点击成员卡会打开页面内 bottom sheet，仅展示姓名、角色、加入时间、当前用户标记和角色说明，不展示 OPENID、内部 userId、requestKey 或完整数据库记录。

## 权限与操作

- owner 点击自己时只显示创建者说明，不显示角色修改、移除或退出入口。
- owner 点击 viewer 时可选择“设为管理员”或“移出团队”。
- owner 点击 admin 时可选择“取消管理员”或“移出团队”。
- admin 和 viewer 只能查看成员详情，不能修改角色或移除成员。
- admin 和 viewer 可在团队页底部主动退出团队。
- owner 不能退出团队；当前版本也不支持 owner 转让或解散团队。

角色变更分别调用 `team.member.role.update`，目标角色只允许 `admin` 或 `viewer`。移除成员调用 `team.member.remove`，本阶段使用固定原因“由团队创建者移出团队”。退出团队调用 `team.leave`。前端不传 teamId、userId、openId 或 ownerId，最终权限仍由 `warehouse-api` 校验。

## requestKey与状态

角色变更、移除和退出分别保存独立操作意图，并通过统一的 `createRequestKey` 生成 requestKey。同一次网络失败或结果不确定的手动重试复用原 key；成功、明确业务失败或用户取消后清除。requestKey 不写 Storage、不显示、不打印，也不在不同 action 之间共用。

团队页分别维护邀请码刷新、成员审核、目标成员操作、退出团队和页面刷新状态。成员操作状态包含目标 memberId；退出使用独立 `leaving`；页面刷新继续使用 Promise 去重。页面卸载后不再执行无意义的 `setData`。

## 状态清理与重新路由

`app.clearTeamContext()` 统一清空 `currentMembership`、`currentTeam`、`currentRole` 和 `currentWarehouse`，但保留当前 user。退出成功后先清理团队上下文，再强制执行 `user.bootstrap`，随后重新进入 startup。

团队页每次 `onShow` 都强制刷新真实 bootstrap。若用户已被移除，或受保护接口返回 `NO_ACTIVE_TEAM`、`MEMBERSHIP_NOT_ACTIVE`、`TEAM_NOT_ACTIVE`，页面会清理旧团队上下文、重新 bootstrap，并由 startup 根据真实申请状态进入 team-setup、team-join 或其他既有流程。当前不实现 WebSocket 实时踢出；状态通过重新进入、页面刷新或接口无权限响应更新。

## 云端边界

阶段2B1已经提供并部署 `team.member.role.update`、`team.member.remove` 和 `team.leave`。本阶段未修改云函数，不新增集合、索引或数据库权限，也不需要重新部署 `warehouse-api`。
