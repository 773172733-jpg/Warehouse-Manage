# 阶段2C1产品、库存与流水架构

## 1. 范围与约束

本阶段只定稿产品、库存余额和库存流水的云端架构，不创建集合、索引，不部署云函数，也不修改阶段1页面。所有业务数据库访问继续经过 `warehouse-api`；客户端不得直接读写业务集合，也不得提供并被信任的 `teamId`、`warehouseId`、`userId`、`operatorId`、`role` 或 `openId`。

设计以当前代码为依据：

- 库存首页使用产品名称、编号和关键词搜索，按分类或库存状态筛选，并展示产品总数、低库存数、缺货数。
- 产品详情展示产品资料、当前库存、最低库存、库存状态和最近流水摘要。
- 新增页支持文字、系统 emoji、自定义图片三种封面，支持初始库存和最低库存。
- 库存操作页支持入库、出库和按目标数量调整，使用原因、来源或去向、备注。
- 流水页支持关键词、类型、产品、操作人和日期筛选，展示不可变的前后库存和快照字段。

## 2. 三种模型比较

| 方案 | 读取与筛选 | 写入一致性 | 多仓库 | CloudBase复杂度 | 结论 |
| --- | --- | --- | --- | --- | --- |
| A：仓库级产品文档 | 单集合即可完成列表、分类、库存状态、排序和统计 | 一份文档内保存余额；事务同时更新产品并写流水 | 同一产品跨仓会产生独立文档 | 最低，不需要联表或二次拼装 | V1采用 |
| B：团队级产品目录 + `inventory_balances` | 列表需要关联或两次查询；搜索和库存排序、统计更复杂 | 余额职责清晰，唯一索引约束仓库产品 | 天然适配共享产品目录 | 中高，主要页面需要跨集合组合 | 当前不采用 |
| C：团队级产品目录 + `warehouse_products` | 若不复制展示字段则仍需关联；复制后可单集合展示 | 需要同步产品主数据和展示快照 | 适合多仓，但存在快照一致性问题 | 最高，兼有B的关联与快照同步成本 | 当前不采用 |

### 2.1 最终决策

V1选择方案A：每个 `products` 文档属于一个团队内的一个仓库，同时保存产品资料、当前库存、最低库存和派生库存状态。

选择理由：

1. 当前只有一个默认仓库，方案A直接匹配现状。
2. 首页是最高频入口，单集合能降低查询、索引、统计和故障处理复杂度。
3. CloudBase虽支持聚合和 `lookup`，但不能按传统SQL联表体验假设所有列表查询都低成本。
4. 产品余额与流水仍通过服务端事务保持一致，不牺牲库存安全。
5. V1先解决真实可用性，不为尚未确认的共享产品目录过度设计。

不选择B，是因为当前每次产品列表、库存筛选和统计都要组合产品资料与余额。即使使用聚合，也会增加索引、分页、错误处理和性能验证工作。

不选择C，是因为不复制名称等字段时与B相同；复制后又必须解决产品改名、分类变化和单位变化的快照同步，复杂度大于V1收益。

### 2.2 多仓库演进

- 只增加多个独立仓库时：继续使用方案A。列表增加经云端校验的仓库选择，现有数据无需迁移。
- 多仓模式下客户端选择的 `warehouseId` 只作为候选参数；云端必须验证该仓库属于可信当前团队且状态为active，再建立请求上下文。
- 需要跨仓共享产品主数据时：迁移到方案B。保留现有产品 `_id` 作为首仓产品目录ID，为每个现有产品生成一条余额文档，再把 `stock`、`minStock`、`stockStatus` 和 `stockVersion` 移入余额集合。
- 历史 `stock_records.productId` 保持不变，产品名称和单位仍由流水快照保证可读，因此无需重写流水。
- 迁移成本为中等：新增余额集合与唯一索引、批量生成余额、双读校验、切换查询接口和清理旧余额字段。迁移需要维护窗口或双写阶段，但不是重做数据。

`lowStockEnabled`只存在于当前本地原型。正式接口不存该字段：关闭提醒等价于提交 `minStock: 0`，确保库存状态始终遵守同一规则。

### 3.2 业务决定

- 产品名称允许重复，产品 `_id` 是唯一身份。编号也不是身份字段，V1允许为空或重复，避免软删除与唯一索引产生复杂恢复规则。
- 分类使用普通字符串；单位使用建议列表加受限自定义字符串，不建立独立集合。
- `code`保留为可选业务编号。V1不新增未使用的SKU或条码字段；确认扫码需求后再新增 `barcode` 及对应索引。
- 库存只支持非负整数，不支持小数和负库存。
- viewer可以看到active产品的展示字段、库存、最低库存、库存状态和时间；不返回团队、仓库、用户内部ID、幂等键或删除审计字段。

## 4. 库存余额与状态

V1没有单独的 `inventory_balances` 集合。等价余额字段位于 `products`：`teamId`、`warehouseId`、`_id`（即productId）、`stock`、`minStock`、`stockStatus`、`stockVersion`、`updatedBy`、`updatedAt`。

库存状态由云端唯一帮助函数计算，并在同一事务中持久化：

- `stock <= 0`：`out`
- `stock > 0 && stock <= minStock`：`low`
- `stock > minStock`：`normal`

持久化 `stockStatus` 是有意的反规范化：它支持索引筛选和三项首页统计。客户端不能提交该字段；每次创建、库存操作或最低库存修改都必须重新计算。可通过离线审计任务比较 `stock`、`minStock` 与 `stockStatus`，发现漂移时报警而不是静默信任。

产品普通编辑不得修改 `stock`。`minStock`属于产品设置，但修改时必须在事务中锁定产品、重算 `stockStatus` 并递增 `stockVersion`。所有实际库存变化只能调用库存接口。

库存写入不要求客户端提交 `stockVersion` 做乐观锁。服务端事务锁定产品文档、在写冲突时有限重试并每次重读最新余额；`stockVersion`用于响应、审计和刷新判断，不替代事务防超卖。

## 5. stock_records集合

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| `_id` | string | 由可信上下文、动作和 `requestKey` 生成确定性ID |
| `teamId`、`warehouseId` | string | 可信服务端上下文 |
| `productId` | string | 产品ID；产品删除后仍保留 |
| `productNameSnapshot` | string | 操作时产品名，历史不随改名变化 |
| `productCodeSnapshot` | string | 操作时编号，可空 |
| `unitSnapshot` | string | 操作时单位 |
| `type` | string | `initial`、`inbound`、`outbound`、`adjust` |
| `changeQuantity` | number | 有符号整数；入库为正、出库为负、调整为目标减原值 |
| `beforeStock`、`afterStock` | number | 非负整数 |
| `reason` | string | 可空；调整必填，最多40字符 |
| `sourceOrDestination` | string | 可空，最多80字符 |
| `remark` | string | 可空，最多200字符 |
| `operatorId` | string | 可信用户ID，仅云端内部使用 |
| `operatorMemberId` | string | 操作时成员关系ID，供验证后的筛选定位 |
| `operatorNameSnapshot` | string | 操作时显示名，历史不随昵称变化 |
| `requestAction` | string | 对应云函数动作 |
| `requestKey` | string | 8至64字符，动作作用域内幂等 |
| `requestInputHash` | string | 规范化业务输入摘要，防止同键异参 |
| `createdAt` | Date | 服务端时间；不可由客户端指定 |

流水创建后不可修改、不可物理删除。产品改名后历史仍显示快照名称；操作人改名或离队后历史仍显示快照。详情可在产品仍active时提供“查看产品”入口，产品删除后只显示快照。

`adjust`允许正差、负差；目标库存等于当前库存属于无效操作，不产生零变化流水。当前mock中的零变化记录仅用于界面健壮性演示，不进入正式写入规则。

## 6. 产品删除与恢复

- 只有owner/admin可软删除。
- `stock > 0`时拒绝删除并返回 `PRODUCT_HAS_STOCK`；必须先通过库存调整归零。
- 删除把 `status` 设为 `deleted`，写入 `deletedAt`、`deletedBy`，保留库存字段和全部流水。
- 重复删除使用相同requestKey返回原成功结果；不同requestKey删除已删除产品返回幂等的已删除摘要，不产生新流水。
- 默认产品列表和详情不读取deleted产品；流水详情始终可读快照。
- 架构允许未来提供owner恢复接口。V1 UI不实现恢复；30天保留期内可由后续接口恢复，之后自定义封面可能已被回收，恢复时退回文字封面。

## 7. 并发、事务与幂等

CloudBase事务只在服务端使用。事务内以确定性文档ID执行 `doc().get/set/update`，不依赖事务内条件查询。事务外鉴权结果只用于快速失败；进入写事务后必须重新读取用户、成员关系、团队、仓库和目标产品，防止权限在提交前变化。

### 7.1 requestKey规则

- 客户端对一次用户意图生成一个8至64字符随机键，网络重试必须复用，用户重新编辑输入后生成新键。
- 创建产品作用域：`teamId + warehouseId + product.create + requestKey`。
- 库存作用域：`teamId + warehouseId + requestAction + requestKey`；流水ID由该作用域确定。
- 服务端保存规范化输入哈希。相同键相同输入返回已有结果；相同键不同输入返回 `DUPLICATE_REQUEST`。
- requestKey、输入哈希和内部确定性ID不返回列表，不写普通业务日志。
- 网络超时后先用同一requestKey重试。库存接口会返回同一流水；产品创建会返回同一产品。更新或删除无法确认时读取 `product.detail`，根据版本和状态确认。

### 7.2 创建产品

1. 验证可信用户、active团队、active默认仓库和owner/admin角色。
2. 校验并规范化产品输入，计算确定性产品ID和可选初始流水ID。
3. 先完成图片上传或使用非图片封面；事务内不调用存储API。
4. 事务内重新读取权限文档及确定性产品文档。
5. 已存在且requestKey、输入哈希一致则返回已有产品；不一致则 `DUPLICATE_REQUEST`。
6. 创建产品，写入初始 `stock`、`minStock`、`stockStatus`、版本和审计字段。
7. 初始库存大于0时，在同一事务创建一条 `initial` 流水，`beforeStock=0`、`afterStock=stock`。
8. 任一步异常抛出，产品和流水全部回滚。初始库存为0时不创建流水。

方案A没有单独余额文档，因此“创建初始库存余额”就是创建产品文档中的余额字段。

### 7.3 入库

1. 校验owner/admin、active团队和仓库。
2. 由请求作用域计算确定性流水ID。
3. 事务内重验权限，读取产品和流水ID。
4. 已有同输入流水则返回已有结果。
5. 校验active产品和正整数数量，计算 `afterStock=beforeStock+quantity`。
6. 重算 `stockStatus`，更新产品库存、`stockVersion`、`updatedBy`、`updatedAt`。
7. 同一事务写入正数 `changeQuantity` 的 `inbound` 流水并提交。

### 7.4 出库

流程与入库相同，但在锁定产品后校验 `quantity <= stock`。`afterStock=stock-quantity`，不足时返回 `INSUFFICIENT_STOCK`，产品和流水均不写入。并发出库因修改同一产品文档发生写冲突，由 `runTransaction` 有限重试；每次重试重新读取最新库存，因此不会超卖。

### 7.5 调整库存

接收非负整数 `targetStock`。事务内锁定产品，计算 `changeQuantity=targetStock-beforeStock`；差值为0时拒绝。更新余额并创建 `adjust` 流水，二者同事务提交。调整原因必填，备注可选。

### 7.6 产品编辑

产品更新携带当前 `version`。事务内读取最新文档；版本不一致返回 `PRODUCT_VERSION_CONFLICT`，前端刷新后让用户确认，不自动覆盖。普通库存操作只递增 `stockVersion`，不制造无关的产品资料冲突。修改最低库存时同时递增 `stockVersion` 并重算状态。

## 8. 图片存储与生命周期

- 本地 `tempFilePath`只用于上传，不写数据库或业务日志。
- 自定义图片上传到轻仓自己的CloudBase存储，数据库只保存fileID，临时HTTPS URL只放接口响应。
- 推荐增加 `product.cover.prepare`：owner/admin请求后由云端返回一次性上传路径和uploadToken；客户端直接上传，创建或更新产品时提交fileID和token，云端验证路径、团队、操作者、过期时间和未绑定状态。
- 列表和详情由 `warehouse-api`批量把已授权的fileID换为临时URL后返回。viewer可读取active产品图片，但无上传、替换和删除权限。
- 每个产品每次版本使用独立图片，不鼓励多个产品共享fileID。删除文件前仍做引用检查。
- 更新成功后旧图片进入延迟回收队列，至少保留7天；产品软删除时保留30天以支持恢复，之后确认无引用再删除。
- 上传成功但业务保存失败的staging文件在24小时后由清理任务回收；客户端取消可主动请求discard。
- 建议仅允许JPEG、PNG、WebP，单张不超过5MB，最长边不超过4096像素，每个产品1张；服务端验证真实MIME、扩展名和大小。
- 存储调用不放进数据库事务。先上传、后事务绑定；数据库成功后再异步清理旧文件。

## 9. 搜索、统计、排序与分页

### 9.1 V1方案

- `normalizedName`和`normalizedCode`由服务端统一生成；`searchText`拼接规范化名称、编号和关键词。
- 名称、编号、关键词参与关键词搜索；分类走精确筛选，介绍不参与搜索。
- 无关键词时，使用 `teamId + warehouseId + status` 前缀索引查询，按 `updatedAt desc, _id desc` 游标分页。
- 有关键词时，V1在可信仓库范围内对转义后的 `searchText`执行受控正则包含查询。CloudBase正则查询不能依赖索引提速，因此限制关键词1至40字符、pageSize不超过20，并把V1建议规模限定为每仓库不超过1000个active产品。
- 首页打开不执行正则。总数、低库存、缺货分别使用带等值条件的count查询并命中状态索引；可在短时间内缓存同一仓库统计，写操作成功后失效。
- 未来按库存排序增加专用复合索引，不复用不匹配的更新时间索引。

### 9.2 扩容方案

当单仓active产品接近1000或搜索P95明显上升时，不继续扩大正则扫描：启用经验证的CloudBase全文索引，或同步到独立搜索服务。切换时保留相同API契约，仅替换服务端搜索实现。不得把正则查询描述成索引全文搜索。

### 9.3 游标

产品游标包含最后一项的 `updatedAt` 和 `_id`；流水游标包含 `createdAt` 和 `_id`。服务端签名或编码游标，客户端不得自行构造排序边界。游标分页避免大offset的扫描成本；V1不提供跳到第N页。

## 10. 索引

字段顺序遵循“可信租户等值条件 -> 业务等值条件 -> 排序字段 -> `_id`稳定次序”。正则搜索不创建无效的 `searchText` 索引。

### 10.1 products

| 索引名 | 字段 | 唯一 | 用途 | 创建时间 |
| --- | --- | --- | --- | --- |
| `idx_products_wh_status_updated` | `teamId asc, warehouseId asc, status asc, updatedAt desc, _id desc` | 否 | active列表、最近更新、总数 | 2C2前 |
| `idx_products_wh_status_stock_updated` | `teamId asc, warehouseId asc, status asc, stockStatus asc, updatedAt desc, _id desc` | 否 | 低库存/缺货筛选与统计 | 2C2前 |
| `idx_products_wh_status_category_updated` | `teamId asc, warehouseId asc, status asc, category asc, updatedAt desc, _id desc` | 否 | 分类筛选 | 2C2前 |
| `idx_products_wh_status_category_stock_updated` | `teamId asc, warehouseId asc, status asc, category asc, stockStatus asc, updatedAt desc, _id desc` | 否 | 分类与库存状态叠加 | 首次实现叠加筛选前 |
| `idx_products_wh_status_name` | `teamId asc, warehouseId asc, status asc, normalizedName asc, _id asc` | 否 | 精确/前缀名称定位，不宣称支持包含正则 | 2C2前 |
| `uidx_products_create_request` | `teamId asc, warehouseId asc, createRequestKey asc` | 是 | 创建幂等第二层约束 | 2C2前 |

产品详情使用默认 `_id` 索引并在读取后校验团队、仓库和状态。方案A没有 `inventory_balances`，因此当前不创建余额唯一索引；未来迁移到方案B时必须创建 `teamId + warehouseId + productId` 唯一索引。

### 10.2 stock_records

| 索引名 | 字段 | 唯一 | 用途 | 创建时间 |
| --- | --- | --- | --- | --- |
| `idx_records_wh_created` | `teamId asc, warehouseId asc, createdAt desc, _id desc` | 否 | 仓库流水、时间范围 | 2C5前 |
| `idx_records_product_created` | `teamId asc, warehouseId asc, productId asc, createdAt desc, _id desc` | 否 | 产品流水摘要与筛选 | 2C2前 |
| `idx_records_type_created` | `teamId asc, warehouseId asc, type asc, createdAt desc, _id desc` | 否 | 类型筛选 | 2C5前 |
| `idx_records_operator_created` | `teamId asc, warehouseId asc, operatorId asc, createdAt desc, _id desc` | 否 | 经成员验证后的操作人筛选 | 2C5前 |
| `idx_records_product_type_created` | `teamId asc, warehouseId asc, productId asc, type asc, createdAt desc, _id desc` | 否 | 产品+类型常用组合 | 2C5前 |
| `uidx_records_request` | `teamId asc, warehouseId asc, requestAction asc, requestKey asc` | 是 | initial及库存写入幂等第二层约束 | 2C2前 |

不为所有筛选排列建立组合索引。V1先覆盖常用路径；操作人、产品、类型全部叠加时由服务端选择最窄索引并在受控扫描预算内继续取数。根据真实慢查询再补索引。

## 11. 数据权限

| 能力 | owner | admin | viewer | pending/removed |
| --- | --- | --- | --- | --- |
| 产品列表/详情 | 允许 | 允许 | 允许 | 拒绝 |
| 新增/编辑/软删除产品 | 允许 | 允许 | 拒绝 | 拒绝 |
| 入库/出库/调整 | 允许 | 允许 | 拒绝 | 拒绝 |
| 流水列表/详情 | 允许 | 允许 | 允许 | 拒绝 |
| 封面临时URL | 允许 | 允许 | 允许active产品 | 拒绝 |

团队必须为active，仓库必须为active，成员关系必须为active。disabled/deleted团队或disabled/deleted仓库拒绝全部业务访问。deleted产品默认不允许详情或库存操作；历史流水通过快照继续读取。

客户端对 `products`、`stock_records` 及未来余额集合保持 `read: false, write: false`。云函数响应使用字段白名单：不返回openId、内部userId、teamId、warehouseId、幂等输入哈希、删除操作者ID或完整数据库文档。

## 12. 官方能力依据

- CloudBase服务端事务支持多文档原子提交和写冲突处理，适合“余额更新+流水写入”；事务内不执行外部存储调用。
- 正则表达式查询不能利用索引，因此V1搜索明确设置规模上限和升级路径。
- 聚合的前置match/sort可利用索引，首页统计应先按可信团队、仓库和状态缩小范围。
- 云存储使用fileID持久化，访问时批量换临时URL，不把临时URL写入数据库。
