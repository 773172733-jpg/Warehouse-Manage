# 阶段2C1产品、库存与流水架构

## 1. 范围与最终约束

阶段2C1只定稿架构，不创建集合、索引，不部署云函数，也不修改阶段1页面。所有产品、仓库产品和流水访问继续经过 `warehouse-api`；客户端不得直接读写业务集合，也不得提供并被信任的 `teamId`、`warehouseId`、`userId`、`operatorId`、`role` 或 `openId`。

负责人已经确认：

- 未来确定需要团队级共享产品目录。
- 每个团队最多99,999个active共享目录产品。
- 产品名称允许重复，产品身份始终使用 `_id`。
- 仓库产品和共享目录产品都必须支持软删除恢复。
- 自定义图片单张最大2MB。
- 库存流水永久保留，用户不能修改或删除；未来只能冷归档。

## 2. 当前原型需求

现有页面实际需要：

- 首页按名称、编号、关键词、分类和库存状态查询，并显示产品总数、低库存数、缺货数。
- 详情展示共享产品资料、当前仓库库存、最低库存、库存状态和最近流水。
- 新增支持文字、系统emoji、自定义图片封面以及初始库存。
- 库存操作支持入库、出库和按目标数量调整，记录原因、来源或去向、备注。
- 流水支持产品、类型、操作人、时间和关键词筛选，展示不可变前后库存与快照。

## 3. 三种模型复核

| 方案 | 优点 | 缺点 | 最终结论 |
| --- | --- | --- | --- |
| A：仓库级产品文档 | 单仓列表最简单 | 同一产品跨仓重复，无法满足已确认的共享目录 | 否决 |
| B：共享目录 + 纯余额 | 职责最纯粹 | 首页每项都需要关联主数据，CloudBase查询和分页复杂 | 不采用 |
| C：共享目录 + 仓库产品快照 | 共享主数据与仓库余额分离；首页可直接使用受控快照 | 需要版本检测和快照同步 | 采用简化方案C |

原方案A建立在“共享目录尚未确认”的前提上。现在该需求已确定，继续使用A会产生跨仓重复产品、资料同步困难和后续高成本迁移，因此不再作为最终架构。

## 4. 最终模型

最终使用三个业务集合：

1. `products`：团队级共享产品主目录，产品资料唯一权威来源。
2. `warehouse_products`：共享产品在具体仓库中的实例、库存余额、阈值和列表快照。
3. `stock_records`：不可变且永久保留的库存流水。

`warehouse-api`负责组合主数据和仓库数据，前端不自行查询或拼接集合。

## 5. products：共享产品主目录

### 5.1 职责

`products`只保存团队级产品资料、封面、搜索字段、版本和生命周期。它不得保存 `warehouseId`、`stock`、`minStock`、`stockStatus` 或 `stockVersion`。

### 5.2 字段

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| `_id` | string | 服务端生成；创建时由可信团队、action和requestKey生成确定性ID |
| `teamId` | string | 可信active团队 |
| `name` | string | 必填，1至40字符，允许重名 |
| `normalizedName` | string | NFKC、去首尾空格、合并空白、转小写 |
| `productCode` | string | 可空，最多40字符，允许重复 |
| `normalizedCode` | string | 服务端规范化编号 |
| `category` | string | 必填，1至20字符 |
| `unit` | string | 必填，1至10字符 |
| `brand` | string | 可空，最多40字符 |
| `specification` | string | 可空，最多80字符 |
| `description` | string | 可空，最多200字符 |
| `searchKeywords` | string[] | 最多10项，每项1至20字符，服务端规范化并去重；客户端禁止提交 |
| `coverType` | string | `text`、`emoji`、`image` |
| `coverText` | string | 文字封面，最多6字符 |
| `coverEmoji` | string | 白名单系统emoji |
| `coverAssetKey` | string | 系统贴图白名单键 |
| `coverFileId` | string | 自定义图片CloudBase fileID，不保存临时URL |
| `coverBackground` | string | 服务端颜色白名单值 |
| `status` | string | `active`、`deleted` |
| `version` | number | 主资料乐观锁版本，从1开始 |
| `activeWarehouseCount` | number | active仓库实例计数，由服务端事务维护 |
| `createRequestKey` | string | 创建幂等键，创建后不可变 |
| `lastMutationAction` | string | 最近主资料更新/目录删除/恢复动作 |
| `lastMutationRequestKey` | string | 最近主资料写入幂等键 |
| `lastMutationInputHash` | string | 最近规范化输入摘要 |
| `createdBy`、`updatedBy` | string | 可信用户ID |
| `deletedBy`、`restoredBy` | string/null | 目录删除和恢复操作者 |
| `createdAt`、`updatedAt` | Date | 服务端时间 |
| `deletedAt`、`restoredAt` | Date/null | 软删除和最近恢复时间 |

团队的 `activeProductCount` 由服务端维护，用于事务性执行99,999上限。创建和目录恢复增加计数，全局目录删除减少计数；客户端不能提交该计数。

### 5.3 规则

- 名称和编号都允许重复，任何写入和跳转都使用 `_id`。
- 分类和单位V1保存受限字符串，不建立独立集合。
- 普通主资料编辑使用 `version` 乐观锁。
- 主资料编辑不得接收任何库存、阈值或库存版本字段。
- products是名称、编号、分类、单位、介绍和封面的最终权威；仓库快照只能加速读取。

### 5.4 searchKeywords 服务端生成规则

searchKeywords 是受控辅助查询字段，只能由 warehouse-api 生成，不承诺十万级任意全文包含搜索。

**生成所有权：**

1. searchKeywords 只能由 warehouse-api 服务端纯函数生成。
2. product.create 和 product.update 统一调用同一个服务端纯函数。
3. 前端不得生成、提交或覆盖 searchKeywords。
4. 即使请求中出现 searchKeywords，接口也必须丢弃并以服务端生成为准。
5. 批量导入、数据修复和后续管理工具必须复用同一生成函数。

**关键词来源（至少包括）：**

- name
- productCode
- category
- brand
- specification

**云端规范化处理：**

- 去除首尾空格
- 统一英文大小写（转小写）
- 规范连续空白（合并为单个空格）
- 去重
- 过滤空值
- 限制关键词数量（最多10项）
- 限制单个关键词长度（1至20字符）

**辅助字段：**

- normalizedName 和 normalizedCode 仍分别保存，支持名称前缀和编号查询。
- searchKeywords 是辅助查询字段，不替代 normalizedName/normalizedCode 的专项查询能力。

**编辑后处理：**

- 产品编辑后必须重新生成 searchKeywords 并更新 version。
- 即使编辑未修改关键词来源字段，仍以当前产品主资料重新生成。
- 重新生成确保关键词始终与产品主资料一致，不受历史残留影响。

**客户端禁止字段（接口契约标明）：**

- `searchKeywords` — 由云端生成
- `normalizedName` — 由云端生成
- `normalizedCode` — 由云端生成

这三个字段即使出现在请求中也丢弃，接口不得信任。

## 6. warehouse_products：仓库实例与库存余额

### 6.1 职责

每个共享产品在一个仓库最多有一条仓库实例。`stock`、`minStock`、`stockStatus`和`stockVersion`只保存在这里。

### 6.2 字段

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| `_id` | string | 由可信teamId、warehouseId、productId生成确定性ID |
| `teamId`、`warehouseId`、`productId` | string | 三者组成唯一关系 |
| `status` | string | `active`、`removed` |
| `stock` | number | 0至999999999的整数 |
| `minStock` | number | 0至999999999的整数 |
| `stockStatus` | string | `out`、`low`、`normal`，云端计算并持久化 |
| `stockVersion` | number | 仓库余额版本，从1开始 |
| `productVersion` | number | 快照对应的products.version |
| `productNameSnapshot` | string | 首页必要快照 |
| `productCodeSnapshot` | string | 首页必要快照，可空 |
| `categorySnapshot` | string | 首页筛选快照 |
| `unitSnapshot` | string | 首页展示快照 |
| `coverSummarySnapshot` | object | type、text、emoji、fileId、background、assetKey的受控子集 |
| `createRequestKey` | string | 首次加入仓库的幂等键 |
| `lastMutationAction` | string | 最近阈值、移除或恢复动作 |
| `lastMutationRequestKey` | string | 最近仓库实例写入幂等键 |
| `lastMutationInputHash` | string | 同键异参检测 |
| `createdBy`、`updatedBy` | string | 可信用户ID |
| `removedBy`、`restoredBy` | string/null | 移除和恢复操作者 |
| `createdAt`、`updatedAt` | Date | 服务端时间 |
| `removedAt`、`restoredAt` | Date/null | 最近移除和恢复时间 |

唯一约束必须是 `teamId + warehouseId + productId`。恢复时复用原文档，不创建第二条实例。

### 6.3 库存状态

- `stock <= 0`：`out`
- `stock > 0 && stock <= minStock`：`low`
- `stock > minStock`：`normal`

`stockStatus`由云端统一帮助函数计算并与余额在同一事务写入。客户端不得提交或修改 `stock`、`stockStatus`、`stockVersion`；`minStock`只能通过仓库产品阈值接口修改。

库存接口在事务内锁定warehouse_products并每次重读最新stock。`stockVersion`用于响应、审计、阈值乐观锁和刷新判断；防超卖最终依赖数据库事务与写冲突重试。

## 7. 主数据快照同步

快照用于首页快速查询，不是最终权威来源。

1. V1只有默认仓库时，`product.update`在同一事务更新products和当前warehouse_products快照，并把productVersion更新为新版本。
2. 多仓库数量处于事务操作上限内时，可在一次事务更新已知仓库实例；事务内仍按确定性文档ID操作。
3. 仓库数量增大后，主事务只提交products和当前仓库实例，其余实例进入异步补偿队列。
4. `product.list/detail`发现 `warehouse_products.productVersion < products.version`时，必须使用products权威字段组成响应，并安全触发快照修复。
5. 补偿失败可重试；任何时候都不得用旧快照反向覆盖products。
6. 产品更新旧图片延迟回收，直到有效仓库快照已更新且确认不存在引用。

无关键词库存列表可直接按warehouse_products索引读取快照；详情始终读取products和warehouse_products。搜索先由products定位候选productId，再由warehouse-api查询并验证当前仓库实例，前端不参与拼接。

## 8. stock_records：永久不可变流水

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| `_id` | string | 由可信团队、仓库和requestKey生成确定性ID |
| `teamId`、`warehouseId` | string | 可信上下文 |
| `productId` | string | 共享目录产品ID |
| `warehouseProductId` | string | 操作时仓库实例ID |
| `productNameSnapshot` | string | 操作时产品名称 |
| `productCodeSnapshot` | string | 操作时编号 |
| `unitSnapshot` | string | 操作时单位 |
| `type` | string | `initial`、`inbound`、`outbound`、`adjust` |
| `changeQuantity` | number | 入库为正、出库为负、调整为目标减原值 |
| `beforeStock`、`afterStock` | number | 非负整数 |
| `reason` | string | 调整必填，最多40字符 |
| `sourceOrDestination` | string | 可空，最多80字符 |
| `remark` | string | 可空，最多200字符 |
| `operatorId` | string | 可信用户ID，仅云端内部使用 |
| `operatorMemberId` | string | 操作时成员关系ID |
| `operatorNameSnapshot` | string | 操作时显示名 |
| `requestAction` | string | 产生流水的action |
| `requestKey` | string | 仓库内写操作幂等键 |
| `requestInputHash` | string | 同键异参检测 |
| `createdAt` | Date | 服务端时间 |

流水创建后永久保留：用户和普通业务接口不能修改或删除；产品、仓库、仓库实例或成员状态变化后，授权用户仍可查询所属团队历史。未来若成本需要，只能将旧记录迁移到不可变冷归档并保留查询和审计链路，不得破坏历史。

## 9. 两层删除与恢复

### 9.1 从当前仓库移除

V1页面中的“删除产品”默认表示移除当前warehouse_products：

1. 只有owner/admin可操作，事务内确认stock为0。
2. 状态改为removed，记录removedAt/removedBy，历史流水不变。
3. 同事务递减products.activeWarehouseCount。
4. 当前仓库回收站必须提供恢复入口。
5. 恢复复用原warehouse_products，stock仍为0，重新复制最新产品快照并递增products.activeWarehouseCount。
6. 产品目录已deleted时，必须先恢复目录产品。
7. 移除和恢复都要求requestKey；同键同输入返回已有结果，同键异参返回DUPLICATE_REQUEST。
8. 仓库移除不删除共享主图。

### 9.2 全局删除共享目录产品

这是未来团队目录管理能力，V1可以不提供UI：

1. 仅owner可操作。
2. products.activeWarehouseCount必须为0，因此所有仓库实例已removed且stock为0。
3. 软删除products并事务性递减teams.activeProductCount。
4. 历史warehouse_products和stock_records永久保留。
5. 必须提供 `product.catalog.restore`；恢复只恢复目录，不自动激活任何仓库实例。
6. 全局软删除不立即删除图片。

所有创建、恢复仓库实例的事务都必须锁定并校验products为active，从而避免与全局删除竞态。

## 10. 创建和库存事务

### 10.1 创建新共享产品

V1单仓库下，`product.create`在同一事务中：

1. 重验用户、成员、团队和默认仓库。
2. 锁定团队并校验activeProductCount小于99,999。
3. 创建products，activeWarehouseCount=1。
4. 创建默认仓库warehouse_products，写入初始stock、minStock、stockStatus和快照。
5. 初始库存大于0时创建initial流水。
6. 增加teams.activeProductCount。
7. 任一步失败全部回滚。

products、warehouse_products和initial流水ID均由可信上下文与requestKey确定。重试同key返回同一结果。未来新增仓库从共享目录选既有product，只创建warehouse_products，不复制products。

### 10.2 入库、出库和调整

- 入库：锁定warehouse_products，增加stock，重算状态、递增stockVersion并创建正数流水。
- 出库：锁定后按最新stock校验；不足返回INSUFFICIENT_STOCK，不写余额或流水。
- 调整：接收非负目标库存，服务端计算差值；目标等于当前值不产生流水。
- 三者都在同一事务完成余额和流水写入，网络超时使用原requestKey重试。

### 10.3 最低库存

`minStock`不属于products。预留 `inventory.threshold.update`，在事务中锁定warehouse_products、校验可选expectedStockVersion、修改minStock、重算stockStatus并递增stockVersion。

## 11. 搜索、99,999规模、统计与分页

### 11.1 两种规模概念

- 存储逻辑上限：每团队最多99,999个active products，由teams.activeProductCount事务性强制执行。
- 搜索性能能力：不等于可以对近10万文档执行任意包含式正则。V1不作这种承诺。

### 11.2 V1搜索

- products负责标准化名称前缀、规范化编号和受控searchKeywords搜索。
- warehouse_products负责当前仓库active/removed、stockStatus和categorySnapshot条件。
- warehouse-api组合候选产品与仓库实例，并返回统一游标。
- 名称前缀使用normalizedName范围查询；编号优先精确匹配normalizedCode；关键词使用规范化精确token。
- 不搜索description，不使用任意包含式正则作为10万规模方案。
- 数据增大或需要拼音、分词、任意包含时，升级为经验证的专用全文索引或搜索服务。

客户端严禁一次性加载全部产品或使用本地全量筛选。列表服务端游标分页，每页默认20、最大50；页面只保留当前需要的数据。

首页统计从warehouse_products使用带索引的count/聚合查询active总数、low和out，并使用短期缓存；仓库产品写入后失效缓存，不在每次打开时无索引扫描全表。

## 12. 图片策略

- 自定义图片仅JPEG、PNG或WebP，单张最大2MB，每个产品1张。
- 客户端上传前初步校验大小和格式，云端绑定前再次验证真实元数据。
- 本地临时路径不入库；数据库只存轻仓CloudBase fileID；临时HTTPS URL只放响应。
- 使用prepare生成短期staging路径和uploadToken，上传后由product.create/update验证绑定。
- 更新主图后延迟删除旧图，先等待仓库快照同步并检查所有有效引用。
- 仓库移除不删除共享主图；全局目录软删除也不立即删除。
- 创建或更新失败的staging孤立文件在24小时后清理。
- viewer可通过读接口获得active产品临时URL，无上传或删除权限。

## 13. 索引设计

字段顺序遵循“租户等值条件 -> 状态/筛选等值条件 -> 排序字段 -> `_id`稳定次序”。以下索引均为规划，尚未创建。

| `uidx_wh_products_request` | `teamId asc, warehouseId asc, createRequestKey asc` | 是 | 加入仓库幂等 |

### 13.3 stock_records

| 索引名 | 字段 | 唯一 | 用途 |
| --- | --- | --- | --- |
| `idx_records_wh_created` | `teamId asc, warehouseId asc, createdAt desc, _id desc` | 否 | 仓库流水和时间范围 |
| `idx_records_wh_product_created` | `teamId asc, warehouseId asc, warehouseProductId asc, createdAt desc, _id desc` | 否 | 仓库产品流水 |
| `idx_records_product_created` | `teamId asc, productId asc, createdAt desc, _id desc` | 否 | 跨仓产品历史 |
| `idx_records_type_created` | `teamId asc, warehouseId asc, type asc, createdAt desc, _id desc` | 否 | 类型筛选 |
| `idx_records_operator_created` | `teamId asc, warehouseId asc, operatorId asc, createdAt desc, _id desc` | 否 | 经验证操作人筛选 |
| `uidx_records_request` | `teamId asc, warehouseId asc, requestKey asc` | 是 | 库存写入幂等 |

不为所有筛选排列预建索引。根据真实慢查询补充必要组合，避免重复索引和写放大。

## 15. 官方能力边界

- CloudBase服务端事务用于多文档原子提交和写冲突重试，事务内不执行文件上传或外部调用。
- 普通正则查询不能依赖索引优化，因此99,999存储上限不等于任意模糊搜索性能承诺。
- 游标分页优于大offset；排序字段后使用 `_id` 保证稳定顺序。
- 云存储持久化fileID，访问时批量换临时URL。
