# 阶段2B2A部署与验收

## 部署说明

本阶段仅接入小程序申请人页面、启动路由和前端 service，不修改 `warehouse-api`，不新增集合、索引或权限，因此无需重新部署云函数。编译前仍需确认 AppID 为 `wxd5819a772c90b7a2`、环境 ID 为 `cloud1-d8gm59cz2be4e7c23`、云函数为 `warehouse-api`。

## 双账号验收

准备已有团队的 owner 账号 A 和无 active 团队的申请人账号 B。

1. A 使用阶段2B1既有测试方式获取 active 邀请码；本阶段尚未接入 owner 邀请码生成 UI。
2. B 从 `pages/startup/startup` 启动，确认没有申请时进入创建团队页，并能点击“加入已有团队”。
3. 输入空值、非法字符、过短邀请码，确认页面只显示友好校验文案且不提交。
4. 分别测试不存在、过期、已撤销和次数耗尽的邀请码，确认输入保留且可重试。
5. 输入有效邀请码，确认按钮提交期间禁用，成功后显示真实团队名称、申请时间和待审核状态。
6. B 关闭并重新进入小程序，确认 startup 自动路由到待审核页，不进入库存 tab。
7. B 连续点击刷新状态，确认没有重复并发请求、白屏或无限 loading。
8. A 通过阶段2B1既有测试方式拒绝申请；B 刷新后应显示申请未通过，并可重新输入邀请码。
9. B 重新申请后由 A 审核通过；B 点击刷新状态，确认重新 bootstrap 并自动进入库存首页。
10. 已有 active 团队的 A 直接打开 team-setup 或 team-join，确认自动回到库存首页。
11. 回归新用户创建团队流程，确认创建成功后仍进入库存页，team.create 的失败重试和 ALREADY_IN_TEAM 恢复正常。

## 错误与隐私检查

控制台和页面不得出现完整邀请码、requestKey、OPENID、用户对象或数据库堆栈。网络中断时只提示检查网络，不进行无限自动重试。pending 页面不得展示其他成员、邀请码或团队内部数据。

## 测试数据清理

只在轻仓环境操作。先记录 B 的用户 ID 和申请对应团队 ID，再精确清理该测试 `team_members` 记录；若 B 已审核通过，同时将 B 的 `users.currentTeamId`、`currentWarehouseId` 清空。测试邀请码是否保留由 owner 决定，不要批量删除 `invites`，不要删除正式 owner、团队或仓库数据。

清理前后都要再次确认当前环境是 `cloud1-d8gm59cz2be4e7c23`。其他项目环境只读检查，不部署、不删除、不修改。
