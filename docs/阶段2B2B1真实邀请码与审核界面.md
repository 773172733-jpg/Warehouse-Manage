# 阶段2B2B1真实邀请码与审核界面

## 问题与目标

阶段1团队页使用 `mock-members.js` 中的固定邀请码和成员数据。模拟邀请码没有对应的 `invites` 云端记录，因此申请人即使输入页面显示的代码，也无法通过 `team.join.apply` 加入真实团队；模拟审核按钮同样不会修改 `team_members`。

本阶段解除团队页对 mock 数据的引用，接入阶段2B1已部署的真实接口。mock 文件保留供历史原型参考，但不再参与团队页运行。

## 真实数据来源

- 当前团队、仓库和角色来自 `user.bootstrap` 后的 `globalData`。
- active成员来自 `team.member.list` 的 `status=active` 查询。
- owner的pending申请来自 `team.member.list` 的 `status=pending` 查询。
- owner邀请码来自 `team.invite.current`；生成或刷新来自 `team.invite.refresh`。
- 批准和拒绝来自 `team.member.review`。

所有页面请求经过 `team-service.js` 和统一 `cloud-service.js`。页面不直连数据库、不直接调用 `wx.cloud.callFunction`，也不会传入 `teamId`、`userId`、`role`或`openId`。

## 邀请码流程

只有真实 bootstrap 角色为 owner 时才查询和显示邀请码区域。没有可用邀请码时页面显示“当前没有可用邀请码”；点击生成后使用24小时有效期、最多20次的默认参数。已有邀请码时刷新前会确认旧邀请码立即失效。

刷新请求的 requestKey 只保存在页面内存。同一次网络或不确定服务错误重试复用原 requestKey，成功后清除，下次主动刷新生成新值。成功响应会先更新页面，再调用 `team.invite.current` 确认云端最终状态。邀请码不会写入 Storage，也不会进入日志。

## 成员和审核流程

owner同时看到真实active成员和pending申请，可以在确认弹窗后批准或拒绝。批准使用viewer角色的后端默认规则，前端不能指定角色；拒绝不增加邀请码使用次数。成功后页面移除对应pending项，并重新查询active、pending和邀请码余量。

admin和viewer只查询、展示真实active成员，不查询或展示邀请码和pending申请。统计值全部由真实返回列表计算，不再使用固定mock数量。

本阶段隐藏尚未真实接入的角色升降级、移除成员、主动退出、备注修改、二维码和分享卡片，避免产生假成功。

## 部署边界

阶段2B1后端接口已经满足本阶段需要，本次未修改 `warehouse-api`。无需重新部署云函数，也无需新增集合、索引或数据库权限。后续仍需接入角色管理、成员移除和退出团队UI。
