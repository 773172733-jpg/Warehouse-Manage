# 阶段2C4A1：首页型号模糊搜索优化部署指南

## 1. 交付范围

本阶段优化库存首页的 `product.list` 搜索，支持产品名称、产品编号/型号
`productCode`、规格 `specification`，并继续保留分类、品牌和原有关键词搜索范围。

搜索仍由 `warehouse-api` 在服务端完成。客户端只保留约400ms防抖、参数传递和结果展示，
不会查询数据库、过滤当前页冒充全库搜索，也不会为每个产品单独调用云函数。

本次不新增集合，不新增或修改索引，不修改数据库或云存储权限，不修改AppID、
CloudBase环境、库存、`stockVersion`、`stock_records`、产品图片或图片清理Worker。

## 2. 规范化与匹配规则

服务端统一使用 `cloudfunctions/warehouse-api/common/product-utils.js` 中的
`normalizeSearchText()`：

1. 转为字符串并执行 `trim`。
2. 执行 Unicode NFKC 规范化，全角英文字母和数字转为兼容半角形式。
3. 英文字母转为小写。
4. 移除半角/全角空格、`-`、`_`、`/`、`\`、`.`、`·` 等常见分隔符。
5. 保留中文、英文字母和数字，不改变剩余字符顺序。

产品字段和用户关键词使用完全相同的规则。例如 `RA1809010`、`ra1809010`、
`RA-180-9010`、`RA 180 9010` 和 `ＲＡ１８０９０１０` 均规范化为
`ra1809010`。

匹配只采用规范化后的连续子串 Token，不使用正则、拼音、错别字纠正、编辑距离或
AI语义搜索。`RA1809010` 可由 `180`、`1809`、`1809010`、`ra180`、
`RA180`、`RA-180` 和 `RA 180 9010` 命中；`1810`、`R901` 或顺序打乱的
字符不会误命中。

## 3. Token与分页策略

继续复用 `products.searchKeywords` 和
`warehouse_products.searchKeywordsSnapshot` 及既有数组索引。仅为
产品名称 `name`、`productCode` 和 `specification` 三个受限字段生成连续片段：

- 最短片段长度：2
- 最长片段长度：20
- 每个源字段最多参与40个字符
- 每个源字段最多40个连续片段
- 单产品全部搜索Token最多136个
- 完整规范化型号优先保留，最长40个字符
- 所有Token去重

产品名称继续保留既有规范化名称前缀查询，同时生成受限连续片段；因此把型号直接填写
在产品名称中的历史用法也可以由内部片段命中。名称、分类、品牌、规格和分词关键词
仍然保留，中文名称搜索不会缩小。

`product.list` 先在 `warehouse_products` 中同时应用当前团队、当前仓库、active状态、
关键词、分类和库存状态，再按 `updatedAt`、`_id` 倒序分页。新游标绑定规范化关键词、
分类、库存状态和排序条件，错误复用到其他条件时返回 `INVALID_CURSOR`。旧格式游标
仍可解析。

## 4. 新建与编辑产品

`product.create` 和 `product.update` 继续由服务端通过 `buildSearchKeywords()`
生成主产品搜索字段和当前仓库快照。新建产品立即获得连续片段；编辑型号后会重建
Token，旧型号片段不再命中，新型号片段可以命中。

型号的真实字段仍是 `productCode`，没有新增第二个型号字段。若历史业务把型号填在
`name` 或 `specification`，这两个字段也会生成同样受限的连续片段。

## 5. 历史产品回填

历史产品原有 `searchKeywords` 没有内部连续片段，因此部署后必须对需要搜索的现有
仓库执行一次批量回填。新增 action 为：

`product.search.rebuild`

该 action 仅允许当前团队的 owner/admin 执行，viewer不能执行。它按当前仓库既有
列表索引读取一批 `warehouse_products`，再通过一次批量 `in` 查询读取对应
`products` 权威主记录，并更新主记录搜索元数据和当前仓库快照。它不会逐产品读取
主记录，不会读取全部团队产品，不会修改产品版本、库存、库存版本、流水、图片或
更新时间。

回填具有幂等性：已是最新搜索字段的记录不会再次写入；中途失败后可从最近成功返回的
游标继续，也可从头重复执行。建议每批20条，允许1至50条。每次响应包含
`scanned`、`updatedProducts`、`updatedWarehouseProducts`、`skipped`、
`hasMore` 和 `nextCursor`。

## 6. 部署与回填步骤

以下步骤需要用户在微信开发者工具中手动完成。本仓库提交不代表正式云端已经部署或
回填。

1. 确认正式AppID为 `wxd5819a772c90b7a2`。
2. 确认正式CloudBase环境为 `cloud1-d8gm59cz2be4e7c23`。
3. 右键现有 `cloudfunctions/warehouse-api`，选择“上传并部署：云端安装依赖”。
4. 不要删除或重新创建同名云函数，不要部署 `product-image-cleanup-worker`。
5. 使用当前团队 owner/admin 账号打开小程序，在开发者工具控制台执行首批调用：

```js
wx.cloud.callFunction({
  name: 'warehouse-api',
  config: { env: 'cloud1-d8gm59cz2be4e7c23' },
  data: {
    action: 'product.search.rebuild',
    requestId: 'search_rebuild_001',
    data: { pageSize: 20 }
  }
}).then(({ result }) => console.log(result && result.data));
```

6. 若返回 `hasMore: true`，把返回的 `nextCursor` 原样放入下一批 `data`：

```js
data: {
  pageSize: 20,
  cursor: '上一批返回的nextCursor'
}
```

7. 重复调用直到返回 `hasMore: false` 和 `nextCursor: null`。
8. 若团队存在多个可切换仓库，应切换到每个需要搜索的仓库后分别执行完整回填。
9. 不要在日志、截图或工单中公开完整业务数据；回填响应仅需核对计数和游标。

## 7. 手动验收

回填完成后，在库存首页验证：

1. 对型号 `RA1809010` 分别搜索 `180`、`1809`、`1809010`、`ra180`、
   `RA180`、`ra1809010`、`RA-180` 和 `RA 180 9010`，均应命中。
2. 对产品名称为 `RA1809100`、编号为空的历史产品搜索 `180`，应命中。
3. 搜索 `1810`、`R901` 和字符乱序内容，不应因本阶段逻辑误命中。
4. 验证中文产品名称、分类、品牌和规格搜索仍正常。
5. 分别组合分类、正常库存、低库存、缺货以及分类加库存状态。
6. 加载下一页，确认无重复、遗漏或条件串页；清空搜索后恢复普通列表。
7. 从详情返回、下拉刷新后，确认仍按当前关键词和筛选重新查询。
8. 用 owner、admin、viewer 搜索，三者均只能看到当前团队、当前仓库的有效产品。
9. 确认 viewer 仍不能新增、编辑、入库、出库或盘点。
10. 回归一次入库和出库，确认库存、`stockVersion` 和永久流水正常。
11. 检查系统贴图、文字封面、自定义图片和私有临时HTTPS图片显示正常。

## 8. 回滚

若搜索代码出现问题，可重新部署本阶段之前确认可用的 `warehouse-api` 版本，并回滚
库存首页提示文案。已经回填的搜索Token是派生元数据，旧版查询可忽略，无需手工删除。

不要回滚或直接修改库存，不要删除 `stock_records`，不要修改产品版本、图片、集合、
索引或权限。若需彻底恢复旧搜索字段，应使用经过审计的服务端批量脚本重新生成，不能
由客户端直接改数据库。

## 9. 已知限制

- 正式环境必须重新部署 `warehouse-api` 后才支持新 action 和新查询逻辑。
- 历史产品必须完成当前仓库回填后，内部型号片段才能被既有数组索引命中。
- 连续片段最短2个字符、最长20个字符；超长型号完整值最多保留40个字符，每字段片段
  数和单产品总Token均有上限，因此不承诺任意超长字段的所有可能子串。
- 本阶段不实现拼音、错别字、乱序、编辑距离、正则或语义搜索。
- 回填只处理当前团队、当前仓库的active产品，不处理回收站记录。
- 全仓旧测试 `stage2c3c1b.test.js` 的旧文档措辞断言，以及静态审计对
  `tests/stage2c3c1c1.test.js` 既有dynamic require的断言，均为开发前遗留，
  与本阶段搜索实现无关。
