# 阶段2B1部署与验收

## 部署前确认

1. 微信小程序AppID必须为`wxd5819a772c90b7a2`。
2. 云环境必须为`cloud1-d8gm59cz2be4e7c23`。
3. 云函数必须为本项目的`warehouse-api`。
4. 任一配置不符立即停止，不要在其他环境创建集合、索引或测试数据。

## 人工配置

1. 在轻仓云开发控制台数据库中新建`invites`集合。
2. 为`invites`创建四个索引：`idx_invites_code_unique`、`idx_invites_team_status_expiry`、`idx_invites_team_request_unique`、`idx_invites_creator_created`，字段、顺序和唯一性见[`../database/indexes.md`](../database/indexes.md)。
3. 确认`team_members`原有三个索引仍可用，并新增`idx_members_user_updated`。
4. 将`invites`权限设置为自定义安全规则`{"read":false,"write":false}`；同时复查其余四个核心集合也是相同规则。
5. 等待索引状态全部变为可用。代码不会自动创建集合、索引或权限。
6. 右键`cloudfunctions/warehouse-api`，选择“上传并部署：云端安装依赖”。确认部署目标仍是轻仓环境。

## 双用户验收

准备owner用户A和未加入团队的用户B。阶段2B1尚未接UI，可在后续2B2页面接入后验收，或通过开发者工具临时调用service；不要把临时代码提交。

1. A调用`team.invite.refresh`，提供新的合法requestKey；确认得到8位邀请码。
2. A调用`team.invite.current`，确认返回同一active邀请码；admin/viewer调用应为`FORBIDDEN`。
3. B调用`team.join.apply`，提供邀请码和requestKey；确认只生成一条pending `team_members`，用户团队指针仍为空。
4. B调用`team.join.status`，只能看到自己的申请和团队简要信息。
5. A调用`team.member.list`，可看到active与pending；admin/viewer只能看到active。
6. A调用`team.member.review`批准B；确认B成为active viewer、用户指针写入、邀请码次数加1。
7. 重复相同审核requestKey，确认不重复增加次数。
8. A将B设置为admin再降为viewer；B和admin尝试改角色应为`FORBIDDEN`。
9. A移除普通成员，确认成员保留为removed且用户指针清空；owner移除自己必须失败。
10. 重新申请后测试reject，确认邀请码次数不增加，用户仍无团队。
11. admin/viewer调用`team.leave`后重新bootstrap，应进入无团队状态；owner退出必须失败。
12. 检查云函数日志只含requestId、action和错误码，不含完整OPENID或用户对象。

## 幂等与权限验收

- 对刷新、申请、审核、角色、移除和退出分别重复相同requestKey，确认结果稳定且无重复副作用。
- 使用不同用户复用相同requestKey，确认不会互相命中。
- 尝试在请求中加入伪造`teamId`、`userId`、`openId`或`role`，确认被拒绝。
- pending、admin和viewer尝试读取邀请码或审核pending，确认返回`FORBIDDEN`。

## 清理测试数据

只在轻仓环境操作。先记录测试团队ID和测试用户ID，再按团队精确筛选：删除测试`invites`；删除或保留测试用户的非owner `team_members`；把测试用户`currentTeamId`、`currentWarehouseId`改为空。不要批量清空集合，不要删除正式owner数据。若要重建整个测试团队，继续遵循阶段2A文档中的仓库、团队和owner关系清理顺序。

最后再次查看云控制台环境ID，确认所有新增`invites`和测试申请只出现在`cloud1-d8gm59cz2be4e7c23`。检查其他环境时只读，不部署、不删除、不修改。
