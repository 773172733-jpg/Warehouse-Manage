# 阶段2C4A：真实库存事务核心部署指南

## 1. 交付范围

阶段2C4A新增以下 `warehouse-api` actions：

- `stock.inbound`：入库
- `stock.outbound`：出库
- `stock.adjust`：盘点调整

同时新增 `miniprogram/services/stock-service.js`，封装
`inboundStock()`、`outboundStock()` 和 `adjustStock()`。本阶段未把这些方法接入正式操作
按钮，也未实现库存流水列表；完整交互属于阶段2C4B。

本次不新增或删除数据库集合，不新增、删除或调整数据库索引，没有修改云存储
权限、AppID、CloudBase环境、图片上传、临时URL或图片cleanup worker。

## 2. 数据模型兼容

库存余额继续使用 `warehouse_products.stock`，最低库存继续使用 `minStock`，状态继续使用
`stockStatus`。独立并发版本继续使用已有的 `stockVersion`，没有使用
`products.version` 代替库存版本。

2C2A及以后创建的 `warehouse_products` 已从 `stockVersion=1` 开始。对于极少数缺失
`stockVersion` 的旧记录，库存事务把当前版本兼容为1；只有在客户端提交
`expectedStockVersion=1` 且事务成功时，才会随余额一起原子写为2。无效、负数或非整数
版本不会被自动修复，而会拒绝操作。该策略不执行批量迁移。

新库存流水在保留旧模型字段的基础上新增或明确写入：

- `type`：`inbound`、`outbound` 或 `adjustment`
- `beforeStock`、`afterStock`、`delta`、`quantity`、`changeQuantity`
- `reason`、`referenceNo`
- `requestAction`、`requestKey`、`requestHash`
- `operatorUserId`、`operatorRole`
- `stockVersionBefore`、`stockVersionAfter`
- 既有产品、仓库、操作人和展示快照字段

历史 `initial` 流水保持原样，不迁移、不重写。`stock_records` 仍是永久不可变流水，
业务接口只允许新增，不提供更新或删除 action。

## 3. 事务与业务规则

每次库存操作在一个 CloudBase 数据库事务中完成：

1. 重新读取并验证用户、团队、成员和当前仓库。
2. 验证成员为 active owner/admin，viewer、pending、removed 均被拒绝。
3. 读取当前 `warehouse_products` 和对应 `products`。
4. 验证仓库实例仍为 active，产品共享目录仍为 active。
5. 比较 `expectedStockVersion` 和当前 `stockVersion`。
6. 计算库存、变化量和统一的 `stockStatus`。
7. 更新 `warehouse_products.stock`、`stockStatus`、`stockVersion` 和审计字段。
8. 新增一条确定性ID的 `stock_records` 流水。
9. 提交事务。

任一步失败都回滚整个事务，不会出现只更新库存、只写流水或只增加版本的部分成功。

计算规则：

- 入库：`afterStock = beforeStock + quantity`
- 出库：`afterStock = beforeStock - quantity`
- 盘点：`afterStock = targetStock`，`delta = targetStock - beforeStock`
- 库存上限：`999999999`
- 库存下限：`0`，不允许负库存
- 状态：`stock <= 0` 为 `out`；否则 `stock <= minStock` 为 `low`；其余为 `normal`

## 4. 幂等与并发

库存事务流水ID由可信的 `teamId`、`warehouseId` 和客户端 `requestKey` 确定性生成，
不依赖新增唯一索引。客户端不能提交 `recordId` 或 `requestHash`。

服务端 `requestHash` 包含：

- action
- warehouseProductId
- quantity或targetStock
- reason
- referenceNo
- expectedStockVersion

同一 `requestKey`、相同 action 和相同参数重试时，返回原流水结果并标记
`idempotent=true`，不再次改变库存或写第二条流水。同一 `requestKey` 被不同 action 或
不同参数复用时返回 `REQUEST_KEY_CONFLICT`。

库存版本不一致返回 `STOCK_VERSION_CONFLICT`，事务不修改库存、不写流水、不增加版本。
两次并发请求使用同一旧版本时，只允许先获得事务提交的一次成功。

## 7. 手动验收

在正式环境准备一个可安全操作的测试产品，记录其
`warehouseProductId`、当前库存和 `stockVersion`。每次请求使用新的合法
`requestKey`；只有重试同一请求时复用原key。

### 入库

以 owner 和 admin 分别调用 `stock.inbound`，提交正整数 `quantity` 和当前
`expectedStockVersion`。确认库存增加、版本加1、状态正确且只新增一条流水。用viewer
调用，确认返回 `FORBIDDEN`。

### 出库与库存不足

调用 `stock.outbound`，确认库存按数量减少。使用等于全部库存的数量，确认结果为0且
状态为 `out`。使用超过当前库存的数量，确认返回 `INSUFFICIENT_STOCK`，库存、版本和
流水数量均不变化。

### 盘点调整

分别把目标库存调高、调低和调为0，确认 `delta` 符号、最终库存和状态正确。盘点必须
填写原因。目标库存等于当前库存时确认返回 `NO_STOCK_CHANGE` 且没有新流水。

### 版本冲突

保留一个旧 `stockVersion`，先用其他请求成功改变库存，再提交旧版本请求。确认返回
`STOCK_VERSION_CONFLICT`，没有部分更新。

### 重复请求

成功请求后原样重发同一 action、参数和 `requestKey`，确认返回同一 `recordId`、
`idempotent=true`，库存只变化一次。保持 `requestKey` 不变但修改数量，确认返回
`REQUEST_KEY_CONFLICT`。

验收期间不要在控制台、截图或普通日志中公开 `requestKey`、完整流水、用户内部ID或
请求哈希。

## 8. 回滚

若部署后发现问题，可重新部署阶段2C4A之前确认可用的 `warehouse-api` 代码版本。
不要删除或修改已经生成的 `stock_records`，不要回退数据库集合或索引，不要恢复
CloudBase权限。若需纠正已经成功提交的真实库存，必须通过经审核的补偿性库存调整保留
审计链，不能直接改余额或删除流水。

前端 `stock-service.js` 本阶段尚未接入正式UI，因此无需回滚页面入口。
