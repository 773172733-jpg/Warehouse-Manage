# 阶段2C3C1 产品图片上传与安全绑定部署指南

本指南已包含阶段2C3C1A私有云存储访问补丁。用户已经完成集合、数据库权限、三个索引和默认私有云存储权限；补丁完成后只需重新部署 `warehouse-api`、清除编译缓存并真实账号验收。代码不会自动删除任何Storage文件。

## 1. 集合（已完成）

正式环境 `cloud1-d8gm59cz2be4e7c23` 已创建：

```text
product_image_assets
```

无需重建该集合。该集合由 `warehouse-api` 自然写入，不要手工插入测试文档。

主要字段如下：

| 字段 | 含义 |
| --- | --- |
| `_id` | 服务端按团队与prepare requestKey确定的资产标识 |
| `teamId`、`createdBy` | 可信团队和创建用户内部ID |
| `stageRequestKey`、`stageRequestHash` | prepare幂等信息 |
| `status` | `awaiting_upload`、`staged`、`bound`、`orphaned`或`rejected` |
| `declaredExtension`、`declaredSizeBytes` | 客户端声明，仅用于预检查和路径后缀 |
| `uploadCloudPath` | 服务端随机生成的临时上传路径 |
| `verifiedCloudPath` | 服务端预生成、不会返回客户端的安全文件路径 |
| `sourceUploadFileId` | 客户端临时上传文件ID，供后续清理 |
| `fileId` | 云函数读取真实字节并重新上传后得到的verified文件ID |
| `detectedMimeType`、`detectedExtension`、`sizeBytes` | 服务端真实字节检测结果 |
| `sha256` | 服务端对真实Buffer计算的完整哈希，仅内部使用 |
| `productId`、`boundBy`、`boundAt` | 产品事务绑定信息 |
| `confirmedAt`、`orphanedAt`、`rejectedAt` | 状态流转时间 |
| `expiresAt`、`cleanupAfter` | 可绑定期限与未来清理时间；本阶段不执行物理清理 |
| `confirmRequestKey`、`confirmRequestHash` | confirm幂等信息 |
| `createdAt`、`updatedAt` | 服务端时间 |

`awaiting_upload` 文档不会写空字符串 `fileId`；该字段只在确认成功后出现。

## 2. 数据库权限（已完成）

`product_image_assets` 已设置为：

```json
{
  "read": false,
  "write": false
}
```

小程序页面和service不得直接读写该集合。只有 `warehouse-api` 使用服务端SDK访问。

### 5. 索引（已完成）

`product_image_assets` 保留以下3个业务索引：

| 索引名 | 字段顺序 | 唯一 |
| --- | --- | --- |
| `uidx_image_assets_stage_request` | `teamId`升序、`stageRequestKey`升序 | 是 |
| `idx_image_assets_product_status` | `teamId`升序、`productId`升序、`status`升序、`updatedAt`降序 | 否 |
| `idx_image_assets_cleanup` | `status`升序、`cleanupAfter`升序、`_id`升序 | 否 |

不创建 `fileId` 唯一索引。awaiting_upload 阶段不写入 fileId；CloudBase 唯一索引会对缺失 fileId 的多条记录产生兼容性冲突，因此本阶段不对 fileId 建立唯一索引。文件归属和绑定一致性由 assetKey、状态机、teamId、productId 及事务校验保证。

这些索引属于新集合，不占用 `warehouse_products` 的单集合20索引额度。不要新增、删除、重排或重建任何 `warehouse_products` 索引。

## 4. Cloud Storage目录与私有权限（已完成）

代码使用两个目录：

```text
product-images/uploads/
product-images/verified/
```

云存储继续保持控制台默认的保守权限：

```text
仅创建者及管理员可读写
```

不需要“所有用户可读”，不需要个人版自定义存储规则，也不需要购买个人版。客户端只向prepare下发的随机uploads路径上传临时文件；`warehouse-api`以服务端权限下载、校验并创建verified文件。

产品接口完成用户、团队、成员关系和产品权限校验后，读取对应bound资产并调用服务端 `cloud.getTempFileURL`，批量返回约1小时有效的HTTPS链接。viewer可以通过这些产品接口查看图片，但不能prepare、confirm、更新或移除封面。客户端不直接读取fileID，也不自行调用临时链接API。

## 5. 文件生命周期

```text
prepare -> awaiting_upload（约30分钟）
confirm -> staged（约24小时可绑定）
product.create/update -> bound
替换或改为非图片封面 -> orphaned（约7天后可清理）
路径、格式、大小或下载校验失败 -> rejected（约24小时后可清理）
```

本阶段只记录 `cleanupAfter`。不要手工或自动删除 `sourceUploadFileId`、旧verified文件、orphaned文件或rejected文件；物理清理由2C3C2处理。

## 6. 部署 warehouse-api

1. 在微信开发者工具确认云环境为 `cloud1-d8gm59cz2be4e7c23`。
2. 右键 `cloudfunctions/warehouse-api`。
3. 选择“上传并部署：云端安装依赖”。
4. 确认云端安装 `wx-server-sdk 4.0.2`。
5. 等待部署完成后，清除微信开发者工具编译缓存并重新编译。
6. 不要修改云函数名称，必须保持 `warehouse-api`。

该SDK版本支持服务端 `downloadFile({fileID})`、`uploadFile({cloudPath, fileContent})`及批量 `getTempFileURL({fileList})`。临时链接请求使用 `maxAge=3600`，链接只存在于API响应和页面短期内存，不写入数据库、产品快照或客户端Storage。

## 7. 创建与编辑验收

1. owner创建产品，选择小于2 MiB的JPG，确认库存卡片和详情显示图片。
2. 分别用PNG和WebP重复创建或替换，确认成功。
3. 选择超过2 MiB、GIF、SVG或改后缀伪装文件，确认被拒绝且没有创建产品。
4. 编辑图片产品只修改名称，确认不重新上传且原图片不变。
5. 替换为新图片，确认新图显示，旧资产状态为orphaned但旧Storage文件仍存在。
6. 图片分别改为文字、系统贴图和默认封面，确认图片字段从产品和仓库快照清空，旧资产变为orphaned。
7. viewer确认没有上传或编辑入口；直接调用prepare/confirm应返回FORBIDDEN。
8. 使用viewer真实账号打开库存首页和详情，确认可经 `warehouse-api` 查看私有图片。
9. 在产品回收站和共享目录回收站确认图片正常显示；人为使图片不可读时应显示默认封面，不白屏。
10. 检查产品接口响应只包含 `imageUrl`、`imageUrlExpiresAt`和`imageAvailable`，不包含fileID、assetKey、Storage路径、SHA-256或资产状态。

## 8. 数据检查

成功创建后检查：

1. `product_image_assets.status=bound`，`productId`等于真实产品ID。
2. `product_image_assets.fileId`路径包含 `/product-images/verified/`。
3. `products.coverType=image`，`coverAssetKey`与资产一致，`coverFileId`等于verified fileID。
4. `warehouse_products.coverSummarySnapshot`同步相同图片。
5. 初始库存大于0时仍只新增一条initial流水；图片流程不新增其他 `stock_records`。
6. 不应在数据库中看到OPENID、本地路径、图片base64或产品完整请求体。

## 9. 常见错误排查

| 错误码 | 优先检查 |
| --- | --- |
| `IMAGE_FILE_PATH_INVALID` | 环境ID是否正确、上传fileID是否精确对应prepare下发路径 |
| `IMAGE_FILE_TYPE_INVALID` | 扩展名、魔数、PNG IEND、WebP RIFF长度或JPEG结束标记 |
| `IMAGE_FILE_TOO_LARGE` | 服务端下载后的真实Buffer是否超过2 MiB |
| `IMAGE_FILE_DOWNLOAD_FAILED` | source文件是否存在、Storage读取权限和环境是否正确 |
| `IMAGE_FILE_CONFIRM_FAILED` | 云函数是否能向verified目录上传Buffer |
| `IMAGE_ASSET_EXPIRED` | prepare或staged是否超过有效期；重新选择图片 |
| `IMAGE_ASSET_ALREADY_BOUND` | 是否错误复用了其他产品的assetKey |
| `IMAGE_ASSET_STATE_CONFLICT` | 是否重复使用rejected/orphaned资产或存在人工改写 |
| 图片显示为默认封面 | 是否已部署最新warehouse-api、资产是否bound、fileID关系是否一致、临时链接单项是否失败 |

控制台日志只应记录action、requestId和错误码，不应打印完整fileID、cloudPath、requestKey、哈希或文件字节。

## 10. 回滚

1. 停止发布包含图片上传入口的小程序版本。
2. 如需回滚 `warehouse-api`，必须回滚到同样支持私有图片临时URL的已验证版本；直接返回cloud fileID的旧版本在当前私有权限下不能保证viewer显示。
3. 不删除 `product_image_assets`、products图片字段或Storage文件。
4. 核对回滚期间没有人工改写资产状态；后续重新部署时继续使用原数据。

## 11. 后续阶段

阶段2C3C2负责按 `status + cleanupAfter + _id` 游标扫描并物理清理过期source、rejected和orphaned文件。本阶段没有Worker、定时任务、自动删除、内容审核、裁剪或压缩服务。
